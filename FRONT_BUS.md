# V1 前端业务功能总结

本文基于 [BUS.md](docs/BUS.md) 中已经确认的 V1 落地能力，整理对应的前端业务功能范围。

本文只描述前端需要承载的业务功能，不讨论技术实现方式。

## 1. 前端整体定位

V1 前端以**内部管理后台**为核心，主要用于：

- 配置业务主数据与接入信息
- 查看订单、支付、履约、账务、通知等业务状态
- 处理异常订单、失败通知、死信任务等人工干预动作
- 支撑“渠道签名下单 -> 支付 -> 供应商履约 -> 结算 -> 通知”的业务闭环验证

V1 的主交易入口仍然是**渠道开放接口**，不是前端页面，因此前端重点是：

- 管理
- 查询
- 监控
- 人工处置

而不是面向终端用户的下单交易页面。

## 2. V1 对应的前端业务功能

### 2.1 登录与权限

前端需要提供基础的后台账号与权限管理能力，包括：

- 管理员登录
- 退出登录
- 后台用户查看
- 后台用户新增
- 角色查看
- 角色新增

这一部分的目标是支撑内部运营和管理人员进入系统并分配基础角色能力。

### 2.2 渠道管理

前端需要提供渠道接入配置与管理能力，包括：

- 渠道主体信息查看与新增
- 渠道接入凭证信息维护
- 渠道商品授权配置
- 渠道价格配置
- 渠道限额配置
- 渠道回调地址与回调签名信息配置

这一部分的目标是让运营或管理人员能够完成渠道接入准备工作，使渠道具备下单、回调和计费所需的基础配置。

### 2.3 商品中心

前端需要提供商品基础资料管理能力，包括：

- 商品分类查看与新增
- 商品查看与新增
- SKU 查看与新增
- SKU 与供应商映射关系配置

这一部分的目标是完成可售商品的基础信息维护，为渠道授权、下单与供应商履约提供统一商品基础。

### 2.4 风控管理

前端需要提供基础风控规则管理能力，包括：

- 风控规则查看
- 风控规则新增
- 黑名单规则维护
- 白名单规则维护
- 金额阈值规则维护

这一部分的目标是支撑 V1 的基础风控策略，让系统能够在下单前完成通过、人工复核、拒绝三类业务判断。

### 2.5 订单中心

前端需要提供订单全流程查询与人工处置能力，包括：

- 订单列表查询
- 订单详情查看
- 订单事件轨迹查看
- 订单主状态查看
- 订单子状态查看
- 订单业务快照查看
- 人工关闭订单
- 标记异常订单
- 追加订单备注

订单中心应作为前端最核心的业务模块之一，支持业务人员围绕单笔订单查看完整交易链路，包括支付、履约、通知、风控等关键信息。

### 2.6 支付管理

前端需要提供支付相关业务查询能力，包括：

- 支付通道查看
- 支付单查看
- 退款记录查看

在 V1 阶段，前端需要能体现以下支付业务结果：

- 余额支付
- 模拟在线支付
- 免支付

这一部分主要用于业务联调、支付状态核对和退款结果查看。

### 2.7 供应商履约管理

前端需要提供供应商基础管理与履约结果查看能力，包括：

- 供应商资料查看与新增
- 供应商配置查看与维护
- 供应商订单记录查看
- 供应商回调日志查看

这一部分的目标是让业务人员能够查看订单提交供应商后的处理过程和回调结果，支撑履约链路的跟踪与问题排查。

### 2.8 结算与账务

前端需要提供基础账务查看能力，包括：

- 账户信息查看
- 账务流水查看
- 分润规则查看
- 分润规则新增

前端应能让业务人员清楚看到 V1 范围内的核心资金动作结果，例如：

- 渠道余额扣减
- 平台资金增加
- 订单成功后的基础记账
- 退款后的资金冲正

### 2.9 通知管理

前端需要提供通知任务查看与补偿能力，包括：

- 通知任务查看
- 通知投递日志查看
- 死信记录查看
- 手工重试通知

这一部分的目标是让业务人员能够确认订单成功或退款成功后的通知是否已投递，以及在失败时执行人工补偿。

### 2.10 Worker 任务管理

前端需要提供异步任务查看与人工处理能力，包括：

- 任务列表查询
- 死信任务查询
- 手工重试任务
- 取消任务

V1 中前端需要重点支撑以下异步任务的可视化与处理：

- 供应商提交任务
- 供应商查询任务
- 通知投递任务

这一部分的目标是让业务人员能够对异步链路进行监控和补偿，避免任务失败后无法追踪。

## 3. 前端需要支撑的业务闭环

从业务使用角度看，V1 前端需要支撑以下完整闭环的配置、查询与人工处置：

1. 配置渠道、商品、SKU、供应商映射、风控规则、回调信息等基础数据。
2. 支撑渠道通过开放接口完成下单。
3. 在后台查看订单创建后的状态流转和事件轨迹。
4. 查看支付结果、履约结果、账务结果、通知结果。
5. 在失败或异常场景下进行备注、异常标记、通知重试、任务重试、任务取消等人工处理。

因此，V1 前端的核心价值不是“发起交易”，而是：

- 管好基础配置
- 看清业务状态
- 支持异常处理
- 支撑联调验证

## 4. V1 前端边界

基于当前 V1 已落地范围，前端暂不需要扩展为以下能力：

- 面向终端用户的商城、下单、支付页面
- 完整的渠道自助门户
- 完整的人工审核工作台
- 复杂多级代理分润与结算单处理页面
- 完整对账闭环页面
- 真实短信、邮件运营能力页面
- 真实微信、支付宝、真实供应商协议接入后的专项运营页面
- 更完整的编辑、停用、审批、导出类后台能力
- 冻结、解冻资金的正式业务操作页面

另外，V1 订单业务应按以下口径理解：

- 以单订单为核心进行查询与处理
- 以单订单单 SKU 模型为主
- 以状态跟踪、异常处理、结果确认为重点

## 5. 结论

如果按 V1 已实现能力来定义前端范围，那么前端应优先建设的是一套**运营与管理后台**，其核心业务价值是：

- 支撑基础业务配置
- 承接订单全链路查询
- 支撑支付、履约、账务、通知的结果查看
- 提供必要的异常处理与补偿入口

前端在 V1 的定位应是：

**“围绕交易闭环的管理、观测、处置型后台”**

而不是：

**“面向终端交易或完整生产运营体系的全量前台/后台产品”**
