## 2. 项目定位

### 2.1 业务定义

本平台是一个面向商户、代理、子代理、企业客户的 ISP 话费充值交易平台，连接上游供应商与下游渠道，提供统一的商品售卖、自动匹配、订单处理、余额扣款、履约、通知、基础账务与风控能力。

V1 的目标不是构建一个“面向所有数字商品的通用交易中台”，而是建设一套可以稳定支撑运营商话费充值的专用交易系统。

### 2.2 V1 建设目标

1. 支持渠道通过开放 API 发起话费充值。
2. 支持平台基于手机号自动识别运营商与省份。
3. 支持平台根据 `mobile + faceValue + product_type` 自动匹配最优商品。
4. 支持渠道余额扣款、失败退款与基础价差记账。
5. 支持多供应商接入、异步履约、回调与轮询补偿。
6. 支持 Webhook 终态通知、日志追踪、异常监控与差异对账。

## 3. V1 范围与明确不做

### 3.1 本期范围

- 运营商话费充值。
- 渠道层级：商户、代理、子代理、企业客户。
- 充值类型：`快充`、`混充`。
- 支付方式：渠道余额扣款。
- 通知方式：Webhook。
- 风控方式：同步 `PASS / REJECT`。
- 供应商能力：商品同步、动态状态同步、订单提单、查单、回调、对账。

### 3.2 明确不做

- 流量充值正文设计与实施。
- 慢充产品。
- 微信、支付宝等在线支付。
- 独立支付服务业务域。
- 多级分润规则引擎。
- 结算单工作流与完整财务对账闭环。
- 短信、邮件通知。
- 人工审核工作台。
- 通用数字商品抽象与跨品类扩展设计。
- 为未来预留但本期没有交付路径的规则引擎、策略引擎。

## 5. 统一业务口径

### 5.1 充值请求口径

开放下单请求固定使用以下业务参数：

- `channelOrderNo`
- `mobile`
- `faceValue`
- `product_type`，可选，默认 `混充`

渠道不得直接传 `skuId` 或内部商品编号。

### 5.2 商品口径

V1 的最小可售单元是 `product_id`，代表一个可以被平台匹配并下单的充值商品。

一个商品必须至少具备以下字段：

- `product_id`
- `product_name`
- `province`
- `ispName`
- `face_value`
- `product_type`
- `purchase_price`
- `amount`
- `sales_status`
- `details`

补充约束：

- `product_type` 仅允许 `快充`、`混充`。
- `sales_status` 仅允许 `下架`、`上架`、`维护中`、`库存维护`。
- `amount` 定义为“上游同步回来的参考可售量”，不是唯一可售依据。

### 5.3 商品唯一键

V1 业务上，一个可售商品由以下业务键唯一确定：

- `province`
- `ispName`
- `face_value`
- `product_type`

其中 `province` 允许两类取值：

- 具体省份，例如“广东”
- `全国`

同一业务键下，不允许存在多个同时 `上架` 的有效商品。若存在，视为配置错误并拒绝下单。

### 5.4 当前实现兼容说明

如果现有代码仍保留 `product + sku` 两层模型，则在 V1 中应以当前 `sku` 级对象等价承载这里定义的 `product_id` 语义。旧的 SPU 仅保留运营归类价值，不再参与对外业务口径。

## 7. 商品匹配与最优路由规则

### 7.1 平台自动识别

平台在接到充值请求后，必须先基于 `mobile` 自动识别：

- 归属省份
- 运营商

渠道上传的请求参数不参与该识别过程。

### 7.2 商品匹配规则

平台按以下顺序匹配商品：

1. 根据手机号识别出的 `province + ispName + faceValue + product_type` 查找省内商品。
2. 若省内商品不存在，再查找 `province=全国` 的全国商品。
3. 若没有找到商品，直接拒单，不创建订单。
4. 若命中商品但渠道未授权、商品非上架、处于维护、库存维护或动态状态失效，则拒单。

### 7.3 最优算法适用边界

V1 中，“最优算法”用于命中商品后的供应商候选路由，而不是用于多个重复商品之间竞争。

商品匹配必须是确定性的；供应商路由允许评分决策。

### 7.4 供应商候选评分规则

在候选供应商均满足启用、验签、动态状态正常、未熔断的前提下，按如下优先级排序：

1. 成功率与稳定性。
2. 成本价。
3. 平台利润。
4. 默认优先级。

若仍并列，则取最近一次成功耗时更低的候选。

## 8. 核心业务主链路

1. 渠道使用签名方式调用开放接口下单。
2. 平台识别手机号归属省份与运营商。
3. 商品服务匹配可售商品并输出候选供应商映射。
4. 渠道服务校验商品授权、渠道售价、限额与回调配置。
5. 结算与账务服务校验余额是否充足。
6. 风控服务同步返回 `PASS` 或 `REJECT`。
7. 订单服务创建订单草稿并固化快照。
8. 结算与账务服务执行余额扣款，扣款成功后订单正式进入 `CREATED`。
9. Worker 触发供应商提单。
10. 供应商服务接收回调或发起查单，持续推进履约。
11. 订单成功则通知服务发送 Webhook。
12. 订单失败或超时则退款并发送终态通知。

| 订单服务 | 下单编排、状态机、超时规则、退款触发、异常标记 | 不做供应商协议适配 |
| 供应商服务 | 商品同步、动态同步、提单、查单、回调、对账、日志 | 不直接改订单主表 |
| 结算与账务服务 | 余额校验、扣款、退款冲正、基础价差记账 | 不做复杂分润和结算单 |

- 余额扣款与退款由订单服务编排、结算与账务服务执行。

## 10. 订单状态与时效规则

### 10.1 主状态

| 状态 | 含义 |
|---|---|
| `CREATED` | 订单已创建，余额已成功扣减，待履约 |
| `PROCESSING` | 已提交供应商，等待受理、回调或查单 |
| `SUCCESS` | 充值成功，订单终态 |
| `FAIL` | 最终失败，准备或等待退款 |
| `REFUNDING` | 退款执行中 |
| `REFUNDED` | 已退款，订单终态 |
| `CLOSED` | 人工关闭，非正常终态 |

### 10.2 子状态

- `supplierStatus`：`WAIT_SUBMIT / ACCEPTED / QUERYING / SUCCESS / FAIL`
- `notifyStatus`：`PENDING / SUCCESS / RETRYING / DEAD_LETTER`
- `refundStatus`：`NONE / PENDING / SUCCESS / FAIL`
- `monitorStatus`：`NORMAL / TIMEOUT_WARNING / MANUAL_FOLLOWING / LATE_CALLBACK_EXCEPTION`

### 10.3 SLA 规则

- 快充：
  - 下单后 10 分钟未成功，进入 `TIMEOUT_WARNING`
  - 下单后 1 小时未成功，自动失败退款
- 混充：
  - 下单后 2.5 小时未成功，进入 `TIMEOUT_WARNING`
  - 下单后 3 小时未成功，自动失败退款

### 10.4 晚到结果规则

- `SUCCESS` 后收到供应商失败，只记异常，不回滚成功。
- `REFUNDED` 后收到供应商成功，打 `LATE_CALLBACK_EXCEPTION`，禁止自动二次扣款。
- 通知失败不阻塞订单进入终态。

## 11. 上游供应商同步、日志与对账

### 11.1 商品同步

供应商服务负责两类商品同步：

1. 全量商品同步  
   每天 1 次，刷新商品目录、产品类型、运营商、省份、供应商商品编码、基础配置。

2. 动态状态/价格/库存增量同步  
   每 60 分钟 1 次，刷新：
   - 上下架/维护状态
   - 采购价
   - 参考库存
   - 通道维护标记

说明：

- 同一上游供应商通道可能同时提供快充与混充，因此动态同步按供应商通道统一执行，不按产品类型拆频率。
- 订单状态同步依赖实时回调与轮询，不依赖商品同步。

### 11.2 动态数据失效保护

当某商品或某供应商映射的动态数据连续两个同步周期未刷新，即超过 120 分钟未更新时，该商品或映射自动转为 `库存维护`，停止参与自动匹配，直到同步恢复或运营手动解除。

### 11.3 实时履约同步

订单履约状态同步采用“回调优先、轮询补偿”：

- 供应商回调实时接入。
- Worker 对未终态订单按产品类型发起查单：
  - 快充：1、3、5、10、20、30、45、60 分钟
  - 混充：5、15、30、60、90、120、150、180 分钟

### 11.4 原始日志与标准日志

供应商服务必须保留两层日志：

1. 原始日志  
   请求报文摘要、响应报文摘要、回调原文、HTTP 状态、耗时、重试次数、验签结果。

2. 标准化日志  
   平台标准状态、标准错误码、标准错误原因、是否需继续轮询、关联订单号、关联供应商订单号。

订单服务只能消费标准化结果，不得直接解析供应商原始报文。

### 11.5 对账任务

V1 必须落地四类对账：

1. 商品数据对账  
   比较平台商品配置与供应商商品目录。

2. 可售状态对账  
   比较平台商品上架状态与供应商实际可售状态。

3. 订单结果对账  
   核对平台订单状态与供应商订单状态。

4. 成本差异对账  
   核对平台记录的采购成本与供应商账单或回执成本。

对账执行节奏建议如下：

- 未终态订单差异扫描：每 10 分钟 1 次
- T+0 全量订单对账：每天 1 次
- T+1 差异复核：每天 1 次

### 11.6 实时降权与熔断

因为动态商品同步频率为 60 分钟，V1 必须增加运行时保护：

- 同一映射在 10 分钟内连续 3 次因库存、维护或协议不可用失败时，自动临时熔断 30 分钟。
- 熔断期间该映射不参与最优路由，不等待下一次同步恢复。

## 12. 非功能与运维要求

- 所有核心链路必须具备请求追踪 ID。
- 所有跨服务事件与重试行为必须可审计。
- 所有终态变更必须保留事件记录，不允许无痕覆盖。
- 供应商回调、轮询、通知、退款都必须幂等。
- 关键日志与对账差异至少保留 180 天。
- 后台必须支持按订单号、手机号、供应商订单号快速定位。

## 13. V1 验收标准

### 13.1 业务验收

1. 渠道只传 `mobile + faceValue` 时，系统默认按混充下单。
2. 渠道传 `product_type=快充` 时，系统只匹配快充商品。
3. 平台可自动识别手机号归属省份与运营商。
4. 无匹配商品、未授权、余额不足、风控拒绝时，系统拒单且不扣款。
5. 正常订单可以完成扣款、履约、成功通知闭环。
6. 快充和混充超时规则分别生效。
7. 失败订单可自动退款并形成冲正流水。
8. 退款后收到晚到成功回调不会自动逆转。
9. Webhook 失败会重试，最终进入死信且不影响订单终态。
10. 供应商商品同步、动态同步、订单对账与差异日志可运行。

### 13.2 文档验收

- 文档不再出现流量充值、慢充、在线支付、短信邮件、人工审核等本期不做能力的设计正文。
- 文档中不再要求渠道传 `skuId`。
- 文档对状态机、时效、退款、同步、对账均给出明确规则，无 `TODO`、`TBD`、远景占位表述。
