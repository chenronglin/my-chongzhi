# 《供应商服务详细 PRD》

## 1. 文档定位

供应商服务是 ISP 话费充值 V1 中最关键、最复杂的模块，负责与上游供应商之间的数据同步、商品映射、提单、查单、回调、日志、对账与健康治理。

## 2. 职责边界

### 2.1 负责内容

- 供应商主数据与凭证管理。
- 商品全量同步。
- 动态状态/价格/库存增量同步。
- 商品与供应商商品编码映射。
- 供应商提单、查单、回调解析。
- 标准状态映射。
- 原始日志与标准日志留存。
- 履约成功率、超时率、错误率统计。
- 未终态订单差异扫描与日对账。
- 供应商通道熔断与恢复。

### 2.2 不负责内容

- 渠道授权与定价。
- 订单主状态写入。
- 渠道余额记账。
- 对渠道直接回调。

## 3. 设计原则

1. 回调优先，轮询补偿。
2. 原始报文与标准结果分层存储。
3. 商品同步与订单同步分离。
4. 对账与日志是核心能力，不是附属能力。
5. 任何供应商协议差异都必须收敛在适配器内部。

## 4. 核心对象

| 对象 | 说明 |
|---|---|
| Supplier | 供应商主数据 |
| SupplierConfig | 凭证、签名、地址、超时配置 |
| SupplierProductSnapshot | 同步回来的供应商商品快照 |
| SupplierProductMapping | 平台商品与供应商商品编码映射 |
| SupplierOrder | 平台向供应商发起的履约记录 |
| SupplierCallbackLog | 供应商回调原文与解析结果 |
| SupplierRequestLog | 提单、查单请求与响应摘要 |
| SupplierReconcileDiff | 对账差异记录 |
| SupplierHealthStat | 成功率、错误率、超时率与熔断状态 |

## 5. 商品同步设计

### 5.1 全量商品同步

执行频率：每天 1 次。

同步内容：

- 运营商
- 省份
- 面额
- 充值类型
- 供应商商品编码
- 商品有效性

输出结果：

- 新商品发现
- 商品下线
- 商品属性变更
- 映射待确认项

### 5.2 动态状态/价格/库存增量同步

执行频率：每 60 分钟 1 次。

同步内容：

- 商品销售状态
- 采购价
- 参考库存
- 维护标记
- 通道开关

关键说明：

- 同一上游渠道可能同时提供快充与混充，动态同步以“供应商通道”为单位执行。
- 不为快充单独拆同步频率。

### 5.3 动态数据失效策略

1. 连续两个同步周期未刷新，即超过 120 分钟无新鲜数据，映射自动转 `库存维护`。
2. 转入 `库存维护` 后，不能继续参与自动匹配。
3. 同步恢复后，需要重新进入有效状态检查。

## 6. 履约执行设计

### 6.1 标准适配器接口

每个供应商适配器必须实现：

- `syncCatalog()`
- `syncDynamicState()`
- `submitOrder()`
- `queryOrder()`
- `parseCallback()`

### 6.2 提单流程

1. Worker 投递提单任务。
2. 供应商服务读取订单快照与候选映射。
3. Route Selector 选出当前最优映射。
4. 调用适配器提单。
5. 记录原始请求、原始响应与标准结果。
6. 若未终态，则等待回调并安排查单任务。

### 6.3 查单节奏

快充查单时间点：

- 1、3、5、10、20、30、45、60 分钟

混充查单时间点：

- 5、15、30、60、90、120、150、180 分钟

### 6.4 状态映射

平台标准供应商状态：

- `ACCEPTED`
- `QUERYING`
- `SUCCESS`
- `FAIL`
- `UNKNOWN`

## 7. 路由与健康治理

### 7.1 候选过滤条件

- 映射状态为 `ACTIVE`
- 商品状态为 `上架`
- 动态状态不是 `库存维护`
- 供应商凭证有效
- 未处于熔断状态

### 7.2 评分顺序

1. 最近成功率
2. 最近稳定性与超时率
3. 采购价
4. 平台利润
5. 默认优先级

### 7.3 实时熔断

满足以下任一条件时，对映射自动熔断 30 分钟：

- 10 分钟内连续 3 次库存不足
- 10 分钟内连续 3 次维护中
- 10 分钟内连续 3 次协议级失败

熔断期间不参与自动路由，支持后台人工解除。

## 8. 日志设计

### 8.1 原始日志

必须记录：

- 请求摘要
- 响应摘要
- 回调原文
- HTTP 状态码
- 耗时
- 验签结果
- 重试次数
- 关联订单号
- 关联供应商订单号

### 8.2 标准日志

必须记录：

- 标准状态
- 标准错误码
- 标准错误原因
- 是否需继续轮询
- 最终路由映射
- 标准成本价

订单服务只消费标准日志结果。

## 9. 对账设计

### 9.1 商品对账

对比平台商品与供应商商品目录，识别：

- 平台有、供应商无
- 供应商有、平台无
- 属性不一致

### 9.2 可售状态对账

对比平台上架商品与供应商实际可售状态，识别：

- 平台仍售卖但供应商已维护
- 平台停售但供应商仍返回正常

### 9.3 订单结果对账

对比平台订单终态与供应商订单终态，识别：

- 平台处理中，供应商已成功
- 平台已退款，供应商后续成功
- 平台成功，供应商查无此单

### 9.4 成本差异对账

对比平台记录的采购成本与供应商账单成本，识别：

- 成功订单成本不一致
- 失败订单仍被供应商计费

### 9.5 对账节奏

- 未终态订单差异扫描：每 10 分钟 1 次
- T+0 全量订单对账：每天 1 次
- T+1 差异复核：每天 1 次

## 10. 接口设计

### 10.1 后台 API

- `GET /admin/suppliers`
- `POST /admin/suppliers`
- `POST /admin/suppliers/:supplierId/config`
- `GET /admin/suppliers/:supplierId/sync-logs`
- `GET /admin/suppliers/reconcile-diffs`
- `POST /admin/suppliers/:supplierId/recover-circuit-breaker`

### 10.2 内部 API

- `POST /internal/suppliers/orders/submit`
- `POST /internal/suppliers/orders/query`
- `POST /internal/suppliers/catalog/full-sync`
- `POST /internal/suppliers/catalog/delta-sync`
- `POST /internal/suppliers/reconcile/orders`

### 10.3 回调 API

- `POST /callbacks/suppliers/:supplierCode`

## 11. 数据设计建议

- `supplier.suppliers`
- `supplier.supplier_configs`
- `supplier.supplier_product_snapshots`
- `supplier.supplier_product_mappings`
- `supplier.supplier_orders`
- `supplier.supplier_request_logs`
- `supplier.supplier_callback_logs`
- `supplier.supplier_reconcile_diffs`
- `supplier.supplier_health_stats`

## 12. 异常处理

- 回调验签失败：记原始日志，不推进订单。
- 查单超时：记日志并由 Worker 重试。
- 商品同步失败：保留旧数据，但若超过 120 分钟无刷新则自动切 `库存维护`。
- 已退款后收到成功回调：产生日志与对账差异，不自动再次扣款。

## 13. 验收标准

1. 支持至少 1 家真实供应商接入。
2. 每天 1 次全量商品同步、每 60 分钟 1 次动态同步可运行。
3. 提单、查单、回调日志完整留存。
4. 未终态订单可通过查单补偿推进。
5. 可输出商品差异、订单差异、成本差异清单。
6. 熔断、恢复、降权规则可验证。

## 14. V1 不做

- 供应商侧自动退款依赖。
- 晚到成功后的自动二次扣款。
- 多供应商账单结算工作流。
