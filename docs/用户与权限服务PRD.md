# 《用户与权限服务详细 PRD》

## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 文档名称 | 用户与权限服务详细 PRD |
| 所属系统 | 交易平台后台 API 服务系统 |
| 模块名称 | 用户与权限服务 / Identity & Access Service |
| 版本 | V1.0 |
| 优先级 | P0 |
| 文档目标 | 明确后台账号体系、登录鉴权、RBAC、数据权限、审计日志与组织边界 |

## 2. 模块背景与定位

### 2.1 模块背景

平台涉及运营、财务、风控、客服、技术支持等多角色协同。如果后台账号体系与渠道主体混在一起，会导致权限混乱、数据越权与审计缺失。

### 2.2 一句话定位

用户与权限服务是平台后台统一身份与访问控制服务，负责后台账号、登录鉴权、RBAC、数据权限与操作审计。

### 2.3 模块价值

1. 统一后台登录身份。
2. 统一角色、权限与数据范围控制。
3. 统一敏感操作审计。
4. 为所有后台服务提供鉴权与权限校验能力。

## 3. 职责边界

### 3.1 用户与权限服务负责什么

1. 后台用户账号管理。
2. 登录、登出、Token 签发与刷新。
3. 角色、权限、数据权限管理。
4. 登录日志与操作审计日志。
5. 部门、岗位、组织信息等后台组织维度管理。

### 3.2 用户与权限服务不负责什么

1. 不维护渠道主体、API Key、渠道价格与授权。
2. 不维护订单、支付、商品、账务等业务事实。
3. 不负责开放平台签名鉴权。
4. 不承载供应商凭证与支付凭证。

### 3.3 与渠道服务边界

- 渠道主体归渠道服务管理。
- 后台账号归用户与权限服务管理。
- 若未来存在渠道门户账号，账号身份归用户与权限服务，账号关联的渠道主体归渠道服务。

## 4. 设计原则

### 4.1 统一身份原则

后台用户必须通过统一身份源登录，不允许业务服务各自维护后台账号。

### 4.2 最小权限原则

默认最小授权，敏感操作需明确角色或审批能力。

### 4.3 数据权限前置原则

数据访问范围需要在服务层统一校验，而不是由页面或调用方自行约束。

### 4.4 审计完整原则

登录、权限变更、敏感操作必须审计，支持追溯操作者、时间、来源与前后值。

## 5. 模块目标与成功标准

### 5.1 业务目标

- 支撑后台多角色协同操作。
- 控制敏感数据访问范围。
- 为审计、合规、风控提供统一身份事实。

### 5.2 技术目标

- 稳定 Token 鉴权。
- 统一权限校验接口。
- 可检索的登录与操作审计。

### 5.3 成功标准

- 所有后台接口都通过统一鉴权。
- 角色权限变更可追踪。
- 敏感操作可定位到人、时间与来源 IP。

## 6. 角色与使用场景

| 角色 | 使用场景 |
|---|---|
| 平台超级管理员 | 管理账号、角色、权限、数据权限 |
| 平台运营 | 访问商品、渠道、订单等后台功能 |
| 平台财务 | 访问结算、账务、退款、对账相关数据 |
| 平台风控 | 访问风险规则、审核与黑白名单 |
| 技术支持 | 排障、日志检索、任务重试 |

## 9. 核心流程设计

### 9.1 登录流程

1. 用户输入账号与密码。
2. 服务校验账号状态与密码。
3. 写登录日志并签发 Token。
4. 返回用户基本信息、角色信息、数据权限摘要。

### 9.2 权限校验流程

1. 后台服务带 Token 调用内部鉴权接口或网关中间件。
2. 用户与权限服务返回用户身份、角色、权限点、数据权限范围。
3. 业务服务结合资源对象进行最终授权。

### 9.3 审计流程

1. 用户执行敏感操作。
2. 业务服务调用审计写入接口或发送审计事件。
3. 审计日志记录操作前后值、操作者、来源 IP、时间与资源标识。

## 10. 功能需求明细

### 10.1 后台账号管理

- 用户新增、编辑、启用、禁用、重置密码。
- 账号锁定与解锁。
- 按部门、岗位、角色检索。

### 10.2 角色与权限管理

- 角色新增、编辑、启停。
- 角色绑定菜单、接口权限、操作权限。
- 预置超级管理员、运营、财务、风控、客服、技术支持等基础角色。

### 10.3 数据权限管理

- 支持全量、部门、自定义、渠道范围等数据权限模式。
- 数据权限与角色分离，可独立配置。

### 10.4 会话管理

- 支持登录、登出、刷新 Token、强制失效。
- 支持查看在线会话与异常会话。

### 10.5 审计与登录日志

- 登录成功、失败、退出日志。
- 敏感操作审计日志。
- 权限变更审计。

## 11. 核心规则

1. 禁用账号不得继续访问后台接口。
2. 锁定账号需明确解锁流程。
3. 超级管理员数量应受控并具备审计。
4. 数据权限不能依赖前端筛选替代后端校验。
5. 审计日志不得因业务删除而丢失。

## 14. 跨服务协作

### 14.1 与所有后台服务

- 提供统一 Token 鉴权与权限校验能力。

### 14.2 与渠道服务

- 后台账号可关联渠道主体，但渠道主体数据由渠道服务维护。

### 14.3 与风控服务

- 高风险登录与敏感操作可输出风险信号。

## 15. 异常处理

### 15.1 连续登录失败

- 触发账号锁定策略。
- 记录登录失败原因与来源 IP。

### 15.2 Token 泄漏或会话异常

- 支持强制失效会话。
- 敏感操作可要求二次确认。

### 15.3 权限配置错误

- 阻断保存高危冲突配置。
- 审计并允许快速回滚到上一个权限版本。

## 16. 权限、审计与非功能要求

### 16.1 权限设计

- 超级管理员拥有全局权限，但需审计。
- 财务、风控、运维等敏感角色需精细化控制。

### 16.2 非功能要求

- 登录接口平均响应小于 200ms。
- 权限校验接口平均响应小于 100ms。
- 审计日志支持长期保留与快速检索。

## 17. 风险点与控制策略

| 风险 | 影响 | 控制策略 |
|---|---|---|
| 账号与渠道混用 | 权限边界混乱 | 严格区分后台身份与渠道主体 |
| 数据越权 | 合规风险 | 数据权限后端校验 |
| 审计缺失 | 无法追责 | 操作审计统一归口 |
| 超级权限滥用 | 安全风险 | 最小权限 + 强审计 |

## 18. 验收标准

### 18.1 功能验收

- 可完成登录、登出、账号管理、角色管理、权限管理、数据权限管理。
- 可查看登录日志与操作审计。

### 18.2 数据验收

- 账号、角色、权限、会话、审计日志结构完整。

### 18.3 技术验收

- 后台服务可通过统一接口完成鉴权与权限校验。
- 审计日志支持按用户、时间、资源检索。

## 19. V1 边界

### 19.1 V1 必做

- 后台账号体系
- Token 鉴权
- RBAC
- 数据权限
- 登录日志与操作审计

### 19.2 V1 暂不做

- 第三方 SSO
- 复杂审批流权限模型
- 可视化权限策略编排
