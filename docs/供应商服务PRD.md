# 《供应商服务详细 PRD》

## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 文档名称 | 供应商服务详细 PRD |
| 所属系统 | 交易平台后台 API 服务系统 |
| 模块名称 | 供应商服务 / Supplier Service |
| 版本 | V1.0 |
| 优先级 | P0 |
| 文档目标 | 明确供应商服务的职责边界、适配器模型、标准接口、状态映射、回调处理、路由与可用性控制能力 |

## 2. 模块背景与定位

### 2.1 模块背景

平台的核心竞争力之一是可以稳定接入多个上游供应商，并在协议差异、回调差异、SLA 差异下仍然向订单服务输出统一结果。如果没有独立供应商服务，订单服务会直接耦合各家协议，造成：

- 订单逻辑与上游协议混在一起。
- 新增供应商成本持续升高。
- 回调与轮询逻辑四处散落。
- 熔断、降级、主备切换难以统一实施。

### 2.2 一句话定位

供应商服务是平台对接上游供应链的协议屏蔽层、状态标准化层与可用性控制层。

### 2.3 模块价值

1. 抽象不同供应商的协议差异。
2. 输出统一的提交、查询、退款、回调解析能力。
3. 将供应商返回映射为平台标准状态。
4. 支持多供应商路由、熔断、降级、主备切换。
5. 将上游复杂性隔离在订单服务之外。

## 3. 职责边界

### 3.1 供应商服务负责什么

1. 供应商主数据与供应商配置管理。
2. 适配器注册、版本管理与调试能力。
3. 运行时路由选择与执行。
4. 提交订单、查询订单、发起退款。
5. 解析供应商回调并标准化输出。
6. 记录供应商订单、回调日志、健康指标。
7. 提供熔断、降级、超时、重试、主备切换能力。

### 3.2 供应商服务不负责什么

1. 不维护平台订单主状态。
2. 不做账务、分润、结算、对账。
3. 不决定渠道价格与渠道授权。
4. 不直接回调下游渠道。
5. 不直接写商品主数据或订单主表。
6. 不承载平台 RBAC 体系。

### 3.3 与其他服务边界

| 服务 | 供应商服务负责 | 对方服务负责 |
|---|---|---|
| 商品服务 | 读取候选供应商静态映射 | 配置 SKU 可接入哪些供应商 |
| 订单服务 | 输出标准化供应商事件 | 维护订单状态与终态判断 |
| Worker 任务系统 | 消费任务并返回执行结果 | 调度异步提交、查询、重试、补偿 |
| 结算与账务服务 | 输出履约成功、失败、退款结果 | 资金记账与冲正 |
| 风控服务 | 输出供应商异常指标 | 风险策略与风险决策 |
| 通知服务 | 无直接投递职责 | 下游通知发送 |

## 4. 设计原则

### 4.1 一供应商一适配器原则

每个供应商都必须通过独立适配器接入，实现统一接口，不允许在主流程中写供应商特例分支。

### 4.2 订单服务无感原则

订单服务只能接收标准状态与标准事件，不感知协议字段与供应商回调细节。

### 4.3 标准状态输出原则

供应商返回的受理、处理中、成功、失败、未知、退款状态都必须映射为平台统一口径。

### 4.4 异步优先原则

提交、查询、回调处理尽量异步化，避免在主下单链路中阻塞。

### 4.5 安全隔离原则

供应商账号、秘钥、签名参数、白名单必须加密存储、最小暴露、严格审计。

### 4.6 稳定性优先原则

必须支持超时、重试、熔断、主备切换、失败降级与健康监控。

## 5. 模块目标与成功标准

### 5.1 业务目标

- 支持多个上游供应商快速接入。
- 保证供应链异常时平台仍可降级运行。
- 缩短新增供应商与切换供应商的成本。

### 5.2 技术目标

- 标准适配器接口稳定。
- 回调与轮询结果统一。
- 供应商执行过程可追踪、可重试、可审计。

### 5.3 成功标准

- 新增供应商不需要修改订单服务状态逻辑。
- 重复回调不会重复推进订单状态。
- 单个供应商异常不会直接拖垮整体履约能力。

## 6. 角色与使用场景

| 角色 | 使用场景 |
|---|---|
| 技术支持 | 查看供应商订单、回调日志、执行失败原因 |
| 供应商运营 | 维护供应商启停、优先级、路由标签、配置 |
| Worker 任务系统 | 调用提交、查询、退款执行接口 |
| 订单服务 | 接收供应商标准事件 |
| 风控服务 | 读取供应商成功率、超时率等健康指标 |

## 7. 总体架构设计

建议由以下能力构成：

- 供应商主数据管理
- 供应商配置与凭证管理
- Adapter Registry
- Route Selector
- Submit / Query / Refund Executor
- Callback Parser
- Health Monitor

标准适配器接口建议如下：

```text
submitOrder(request)
queryOrder(request)
refund(request)
parseCallback(request)
```

## 8. 标准对象与状态模型

### 8.1 核心对象

| 对象 | 说明 |
|---|---|
| Supplier | 供应商主数据 |
| SupplierConfig | 供应商配置、凭证、签名参数 |
| SupplierOrder | 向供应商发起履约后的执行记录 |
| SupplierCallbackLog | 供应商回调原文与解析结果 |
| SupplierHealthStat | 成功率、超时率、错误率等健康指标 |

### 8.2 平台标准供应商状态

- `ACCEPTED`
- `PROCESSING`
- `SUCCESS`
- `FAIL`
- `REFUNDING`
- `REFUNDED`
- `UNKNOWN`

### 8.3 状态映射原则

1. 同步提交成功但未终态，映射为 `ACCEPTED` 或 `PROCESSING`。
2. 同步明确成功，映射为 `SUCCESS`。
3. 同步明确失败，映射为 `FAIL`。
4. 无法判断时映射为 `UNKNOWN`，并交由 Worker 查询补偿。
5. 回调与轮询结果统一进入同一标准映射器。

## 9. 路由与执行流程

### 9.1 候选供应商来源

- 订单中固化的候选供应商快照来自商品服务。
- 运行时可结合供应商健康度、熔断状态、时间窗做过滤。

### 9.2 执行前过滤条件

- 供应商启用状态
- 适配器可用状态
- 供应商凭证有效
- 时间窗有效
- 熔断状态允许

### 9.3 路由策略

V1 建议支持：

- 固定优先级
- 主备切换
- 权重路由
- 熔断后自动跳过

### 9.4 履约主流程

1. Worker 任务系统下发履约任务。
2. 供应商服务读取订单快照与候选路由。
3. 通过 Route Selector 选出本次执行供应商。
4. 调用对应 Adapter 的 `submitOrder`。
5. 记录 `supplier_orders` 执行结果。
6. 若未终态，则等待回调或发起轮询。
7. 统一输出标准事件给订单服务。

## 10. 功能需求明细

### 10.1 供应商管理

- 支持供应商新增、编辑、启停、删除保护。
- 支持供应商类型、协议类型、结算信息、联系人信息维护。

### 10.2 配置管理

- 支持凭证、签名算法、地址、超时、白名单、回调秘钥配置。
- 配置变更必须审计。

### 10.3 提交订单

- 接收订单上下文与候选供应商快照。
- 调用适配器并记录请求、响应摘要、执行耗时。
- 同步失败可按路由规则切换下一候选。

### 10.4 查询订单

- 用于无回调、状态不明确、补偿确认等场景。
- 轮询策略由 Worker 任务系统调度，供应商服务只执行查询。

### 10.5 发起退款

- 接收退款指令，调用适配器发起退款。
- 输出标准退款状态给订单服务与结算与账务服务。

### 10.6 回调解析

- 支持多格式回调输入。
- 验签通过后落原始回调日志。
- 解析并映射为标准状态，再发往订单服务。

### 10.7 健康监控

- 统计成功率、失败率、超时率、回调延迟、平均耗时。
- 支持按供应商、商品、适配器版本维度统计。

### 10.8 熔断与降级

- 支持按错误率、连续失败次数、超时率触发熔断。
- 熔断期间停止路由至对应供应商。
- 熔断恢复后需要冷启动观察期。

## 11. 核心规则

1. 同一供应商协议差异必须收敛在适配器内部。
2. 订单服务只接收标准结果，不接触供应商原始报文。
3. 回调日志必须保留原文摘要与解析结果。
4. 同一供应商订单的重复回调必须幂等。
5. 供应商服务记录执行事实，不负责订单终态判定。
6. 退款结果必须同步输出给订单服务与结算与账务服务。

## 13. 数据模型设计

### 13.1 建议表

| 表名 | 用途 |
|---|---|
| `suppliers` | 供应商主表 |
| `supplier_configs` | 供应商配置与凭证 |
| `supplier_orders` | 供应商订单执行记录 |
| `supplier_callback_logs` | 供应商回调日志 |
| `supplier_health_stats` | 供应商健康指标统计 |

### 13.2 关键字段要求

- `supplier_orders` 必须保留 `orderNo`、`supplierId`、`supplierOrderNo`、请求摘要、响应摘要、标准状态、耗时、重试次数。
- `supplier_callback_logs` 必须保留原始请求头、请求体摘要、验签结果、解析结果、幂等键。
- `supplier_health_stats` 必须保留成功率、失败率、超时率、平均回调延迟。

## 14. 跨服务协作

### 14.1 与商品服务

- 商品服务提供候选供应商静态映射。
- 供应商服务负责运行时筛选与执行。

### 14.2 与订单服务

- 供应商服务输出标准事件。
- 订单服务负责状态机推进与终态判断。

### 14.3 与 Worker 任务系统

- Worker 任务系统负责调度 submit/query/refund。
- 供应商服务负责执行与返回标准结果。

### 14.4 与结算与账务服务

- 供应商服务输出履约与退款结果。
- 结算与账务服务据此做记账与冲正。

## 15. 异常处理

### 15.1 提交超时

- 标记为待确认。
- 交由 Worker 查询补偿，不直接判失败。

### 15.2 提交失败

- 若路由策略允许，切换下一候选供应商。
- 若无候选可用，则输出失败事件。

### 15.3 查无此单

- 记录异常并根据业务配置继续重试或转人工。

### 15.4 回调签名失败

- 拒绝处理。
- 保留原始日志并告警。

### 15.5 晚到终态回调

- 仍需记录日志。
- 输出异常标记给订单服务，不直接要求覆盖现有终态。

## 16. 权限、审计与非功能

### 16.1 权限设计

- 供应商配置维护仅限高权限运营或技术支持。
- 凭证查看与测试接口需要额外审计。

### 16.2 非功能要求

- 提交与查询执行过程必须记录耗时。
- 回调接口支持高并发。
- 凭证与回调秘钥加密存储。

### 16.3 审计要求

- 供应商启停、配置变更、适配器切换必须记录审计。
- 支持按 `supplierId`、`supplierOrderNo`、`orderNo`、`requestId` 检索。

## 17. 风险点与控制策略

| 风险 | 影响 | 控制策略 |
|---|---|---|
| 供应商协议差异大 | 接入成本高、维护困难 | 适配器模式与标准接口 |
| 回调不稳定 | 订单状态延迟 | 回调幂等 + Worker 轮询补偿 |
| 单供应商故障 | 履约中断 | 主备、权重、熔断、降级 |
| 凭证泄漏 | 安全风险 | 加密存储、最小暴露、审计 |

## 18. 验收标准

### 18.1 功能验收

- 可完成供应商新增、配置、提交、查询、退款、回调解析全流程。
- 可支持一个 SKU 对接多个供应商，并按规则切换。
- 可输出标准状态给订单服务。

### 18.2 数据验收

- 供应商订单、回调日志、健康指标完整可查。
- 执行记录与回调日志具备一致的关联键。

### 18.3 技术验收

- 适配器接口稳定。
- 回调处理具备幂等。
- 熔断与降级策略可生效并可观测。

## 19. V1 边界

### 19.1 V1 必做

- 供应商主数据与配置
- 一供应商一适配器
- 提交、查询、退款、回调解析
- 标准状态映射
- 健康监控基础指标
- 熔断与主备切换基础能力

### 19.2 V1 暂不做

- 智能路由评分
- 自学习最佳供应商推荐
- 多地域多活协议编排
