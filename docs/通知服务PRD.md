# 《通知服务详细 PRD》

| 项目 | 内容 |

## 2. 模块背景与定位

### 2.1 模块背景

订单终态、退款、异常等结果需要可靠通知给渠道或外部接收方。若通知逻辑散落在订单、支付或供应商服务中，会导致回调不一致与重试策略失控。

### 2.2 一句话定位

通知服务是平台统一的外部投递服务，负责 Webhook、短信、邮件通知及其重试、死信与投递审计。

### 2.3 模块价值

1. 统一对外回调口径。
2. 统一签名、重试与失败处理。
3. 统一通知日志与人工补发能力。

## 3. 职责边界

### 3.1 通知服务负责什么

1. Webhook、短信、邮件通知任务创建。
2. 通知签名、投递、重试与死信处理。
3. 通知日志与投递结果输出。

### 3.2 通知服务不负责什么

1. 不决定订单主状态。
2. 不维护渠道主体或渠道回调配置事实。
3. 不做支付、供应商、账务处理。

## 4. 设计原则

### 4.1 投递与业务分离原则

订单服务只负责触发通知，不负责实际发送。

### 4.2 失败可重试原则

通知失败需标准化重试，不允许业务服务各自实现。

### 4.3 快照投递原则

通知使用订单创建时固化的回调配置快照，而不是读取实时配置覆盖历史订单。

## 5. 模块目标与成功标准

### 5.1 业务目标

- 稳定向渠道推送订单结果。
- 支持短信、邮件等补充通知。

### 5.2 技术目标

- 投递结果可追踪。
- 失败可重试。
- 死信可人工补发。

### 5.3 成功标准

- 通知任务可完整记录状态变化。
- 重试规则统一生效。

## 9. 核心流程设计

### 9.1 Webhook 通知流程

1. 订单服务触发通知任务。
2. 通知服务读取回调配置快照与模板。
3. 生成签名并发起投递。
4. 根据响应结果决定成功、失败或重试。

### 9.2 重试流程

1. 首次失败后进入重试计划。
2. Worker 任务系统按退避策略调度重试。
3. 超过阈值后进入死信。

## 10. 功能需求明细

### 10.1 Webhook 通知

- 支持订单终态、退款结果等业务回调。
- 支持签名、超时、重试、响应码校验。

### 10.2 短信与邮件通知

- 支持基础模板化发送。
- V1 以平台内部预警与补充通知为主。

### 10.3 日志与死信

- 记录请求时间、响应时间、状态码、响应摘要。
- 支持按订单号、通知任务号检索。

## 11. 核心规则

1. 通知任务必须幂等创建。
2. 重试必须通过 Worker 执行。
3. 已成功任务不得被普通重复请求覆盖。

## 14. 跨服务协作

### 14.1 与订单服务

- 订单服务触发通知。
- 通知服务回传通知成功或失败事件。

### 14.2 与渠道服务

- 渠道服务提供回调配置事实。
- 通知服务使用订单快照中的配置执行投递。

### 14.3 与 Worker 任务系统

- Worker 任务系统负责通知重试与死信重放。

## 15. 异常处理

### 15.1 回调地址不可达

- 记录失败并进入重试计划。

### 15.2 重复通知请求

- 幂等忽略。

### 15.3 长期失败

- 进入死信并支持人工补发。

## 16. 权限、审计与非功能要求

### 16.1 权限要求

- 人工补发和模板修改需审计。

### 16.2 非功能要求

- 投递日志长期保留。
- 支持高并发回调。

## 17. 风险点与控制策略

| 风险 | 影响 | 控制策略 |
|---|---|---|
| 通知失败无人处理 | 下游状态不同步 | 重试与死信 |
| 配置误变更影响历史订单 | 回调错发 | 快照投递 |
| 重复通知 | 渠道误处理 | 幂等任务 |

## 18. 验收标准

### 18.1 功能验收

- 可完成 Webhook 发送、失败重试、死信补发。

### 18.2 数据验收

- 通知任务、日志、死信结构完整。

### 18.3 技术验收

- 重试机制与 Worker 配合生效。
- 通知结果可回传订单服务。

## 19. V1 边界

### 19.1 V1 必做

- Webhook
- 重试
- 死信
- 基础短信与邮件

### 19.2 V1 暂不做

- 复杂模板市场
- 多渠道营销触达编排
