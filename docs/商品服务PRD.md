# 《商品服务详细 PRD》

## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 文档名称 | 商品服务详细 PRD |
| 所属系统 | 交易平台后台 API 服务系统 |
| 模块名称 | 商品服务 / Product Service |
| 版本 | V1.0 |
| 优先级 | P0 |
| 文档目标 | 明确商品服务的职责边界、标准商品模型、SKU 模型、可售规则、静态供应商映射与对内对外接口设计 |

## 2. 模块背景与定位

### 2.1 模块背景

平台需要同时面向多个上游供应商与多个下游渠道输出统一商品能力。如果没有独立商品服务，常见问题包括：

- 同一业务商品在多个系统中定义不一致。
- 订单创建时无法固化稳定的商品快照。
- 渠道授权、展示与价格策略难以统一。
- 上游供应商切换需要直接改订单或渠道逻辑。

### 2.2 一句话定位

商品服务是平台的标准商品服务，负责维护 SPU、SKU、分类、属性、上下架与静态供应商映射，并为订单、渠道、供应商等服务提供统一商品事实。

### 2.3 模块价值

1. 统一平台商品语言与 SKU 口径。
2. 为订单服务提供可售校验与快照来源。
3. 为渠道服务提供授权商品基座。
4. 为供应商服务提供候选供应商静态映射。
5. 为结算与账务服务提供成本和商品属性参考。

## 3. 职责边界

### 3.1 商品服务负责什么

1. 商品分类、SPU、SKU 与属性模型管理。
2. 商品上下架与有效期管理。
3. 商品基础可售规则管理。
4. SKU 与供应商的静态映射配置。
5. 商品快照输出与内部查询能力。
6. 商品变更审计、导入导出与检索。

### 3.2 商品服务不负责什么

1. 不创建订单、不维护订单状态。
2. 不做支付、退款、账务、分润处理。
3. 不直接调用供应商协议。
4. 不维护渠道价格、限额、API Key、白黑名单。
5. 不做风控决策与通知发送。
6. 不承担运行时库存或卡密分配动作。

### 3.3 与其他服务边界

| 服务 | 商品服务负责 | 对方服务负责 |
|---|---|---|
| 订单服务 | 提供 SKU 可售判定与订单快照 | 创建订单、状态流转、事件管理 |
| 渠道服务 | 提供标准商品与全局上下架事实 | 渠道授权、渠道价格、渠道限额 |
| 供应商服务 | 提供静态供应商映射与商品属性 | 运行时路由、适配器调用、状态映射 |
| 结算与账务服务 | 提供商品类型、面额、基础成本参考 | 分润、收支、结算、对账 |
| 风控服务 | 提供商品类别、面额、运营商等信号 | 风险规则与风险决策 |
| Worker 任务系统 | 提供静态映射读取能力 | 任务调度、重试、补偿 |

## 4. 设计原则

### 4.1 标准化原则

平台只能基于标准商品与标准 SKU 对外售卖，不允许直接暴露供应商原始商品结构。

### 4.2 静态配置与动态执行分离

- 商品服务维护 SKU 对应的候选供应商静态映射。
- 供应商服务在运行时决定具体提交与重试路径。

### 4.3 快照优先原则

订单创建时必须固化商品快照、SKU 快照、候选供应商快照与基础价格快照，后续配置变更不影响历史订单解释。

### 4.4 全局可售与渠道可售分离原则

- 商品服务只判断全局可售。
- 渠道服务负责渠道维度的授权、价格、限额与特殊禁售。
- 订单创建时需要同时通过商品服务与渠道服务校验。

### 4.5 多供应商兼容原则

一个 SKU 可绑定多个供应商映射，需支持优先级、权重、时间窗、默认路由与失效控制。

## 5. 模块目标与成功标准

### 5.1 业务目标

- 让所有订单都能基于统一商品事实创建。
- 让渠道看到的商品集合可以稳定授权与管理。
- 让供应商切换不影响下游商品定义。

### 5.2 技术目标

- 支持快速查询可售 SKU。
- 支持稳定输出订单快照。
- 支持静态映射配置审计。

### 5.3 成功标准

- 任意订单都能查询到创建时商品快照。
- 商品变更不会影响历史订单解释。
- 单一 SKU 可支持多个供应商候选映射。
- 商品上下架、失效、禁售规则有清晰审计记录。

## 6. 角色与使用场景

| 角色 | 使用场景 |
|---|---|
| 平台运营 | 维护商品分类、SPU、SKU、上下架与展示配置 |
| 商品运营 | 维护商品属性、面额、有效期、排序、标签 |
| 技术支持 | 查询商品快照、定位映射错误、导出配置 |
| 渠道服务 | 获取某渠道可授权的标准商品基座 |
| 订单服务 | 校验 SKU 可售并生成订单快照 |
| 供应商服务 | 获取 SKU 候选供应商静态映射 |

## 7. 核心对象模型

### 7.1 商品层级

| 对象 | 说明 |
|---|---|
| 分类 Category | 商品分类树，用于组织和检索 |
| SPU | 标准商品定义，例如话费充值、流量包 |
| SKU | 可售最小单元，例如运营商 + 地区 + 面额 |
| SKU 供应商映射 | SKU 对接的候选供应商静态映射 |

### 7.2 商品核心字段

SPU 关键字段建议包括：

- `productId`
- `categoryId`
- `productName`
- `productType`
- `deliveryType`
- `targetType`
- `status`
- `validFrom` / `validTo`
- `baseAttributes`
- `sortNo`

SKU 关键字段建议包括：

- `skuId`
- `productId`
- `skuName`
- `faceValue`
- `operator`
- `region`
- `saleStatus`
- `baseCostPrice`
- `baseSalePrice`
- `limitRuleSnapshotVersion`

SKU 供应商映射关键字段建议包括：

- `mappingId`
- `skuId`
- `supplierId`
- `supplierSkuCode`
- `priority`
- `weight`
- `routeType`
- `costPrice`
- `validFrom` / `validTo`
- `status`

## 8. 状态模型与可售规则

### 8.1 商品状态

- `DRAFT`
- `ACTIVE`
- `INACTIVE`
- `EXPIRED`

### 8.2 SKU 状态

- `INIT`
- `ON_SHELF`
- `OFF_SHELF`
- `LOCKED`
- `EXPIRED`

### 8.3 全局可售判定规则

SKU 全局可售需同时满足：

1. SPU 状态为 `ACTIVE`。
2. SKU 状态为 `ON_SHELF`。
3. 当前时间处于商品有效期内。
4. 至少存在一条生效中的候选供应商映射。
5. 商品未被全局禁售或锁定。

注意：渠道授权、渠道价格、渠道限额不在商品服务内判定。

## 9. 核心流程设计

### 9.1 商品维护流程

1. 创建分类。
2. 创建 SPU，定义商品类型、交付方式、目标类型。
3. 创建一个或多个 SKU。
4. 维护 SKU 属性与基础价格。
5. 配置候选供应商静态映射。
6. 完成审核或校验后上架。

### 9.2 订单创建前查询流程

1. 订单服务传入 `skuId`、渠道上下文、业务参数。
2. 商品服务返回全局可售结果。
3. 同时返回商品快照与候选供应商映射快照。
4. 订单服务将快照写入订单表。

### 9.3 静态映射维护流程

1. 为 SKU 选择一个或多个供应商。
2. 配置优先级、权重、有效期、成本价、路由标签。
3. 失效旧映射时保留历史记录，不影响已下单订单。

## 10. 功能需求明细

### 10.1 分类管理

- 支持分类新增、编辑、排序、停用。
- 支持多级分类，但 V1 建议控制在 2 至 3 级。
- 已关联商品的分类不可直接删除。

### 10.2 SPU 管理

- 支持标准商品创建、编辑、复制、停用、过期。
- 支持商品标签、描述、展示文案、交付说明。
- 支持按分类、商品类型、交付方式筛选检索。

### 10.3 SKU 管理

- 支持面额、运营商、地区、套餐类型等维度建模。
- 支持批量上架、下架、锁定、解锁。
- 支持 SKU 级基础价格与成本价维护。

### 10.4 商品上下架

- 上架前必须完成基础字段校验。
- 下架后禁止新订单，但历史订单仍可查询与追溯。
- 锁定用于紧急止损，优先级高于正常上下架。

### 10.5 属性与规则管理

- 支持商品级属性模板与 SKU 级属性扩展。
- 必须明确哪些字段用于展示，哪些字段用于履约、风控、结算。
- 属性变更需写审计日志。

### 10.6 SKU 供应商静态映射管理

- 支持一个 SKU 绑定多个供应商。
- 支持主备、优先级、权重、有效期、路由标签。
- 映射停用采用状态失效，不做物理删除。

### 10.7 查询与快照输出

- 支持后台查询、开放查询、内部查询三类读能力。
- 支持输出商品快照、SKU 快照、候选供应商快照。
- 支持批量查询与导出。

### 10.8 审计与导出

- 审计记录必须包含操作人、变更前后值、变更原因、时间。
- 支持按商品、SKU、分类、供应商映射维度导出。

## 11. 核心规则

1. 商品、SKU、映射默认禁止物理删除。
2. 历史订单引用过的商品快照必须永久可查。
3. 同一时间窗内，同一 SKU 与同一供应商的生效映射不应产生歧义。
4. 商品服务只给出全局可售事实，不直接返回渠道是否允许下单的最终结论。
5. 商品基础价格可作为参考，不直接等于渠道成交价。
6. V1 不在商品服务内实现复杂促销、库存编排或动态路由决策。

## 12. API 设计

### 12.1 后台管理 API

- `GET /admin/product-categories`
- `POST /admin/product-categories`
- `PUT /admin/product-categories/{categoryId}`
- `GET /admin/products`
- `POST /admin/products`
- `PUT /admin/products/{productId}`
- `POST /admin/products/{productId}/activate`
- `POST /admin/products/{productId}/deactivate`
- `GET /admin/product-skus`
- `POST /admin/product-skus`
- `PUT /admin/product-skus/{skuId}`
- `POST /admin/product-skus/{skuId}/on-shelf`
- `POST /admin/product-skus/{skuId}/off-shelf`
- `GET /admin/product-supplier-mappings`
- `POST /admin/product-supplier-mappings`
- `PUT /admin/product-supplier-mappings/{mappingId}`

### 12.2 开放平台 API

- `GET /open-api/products`
- `GET /open-api/products/{productId}`
- `GET /open-api/products/{productId}/skus`
- `GET /open-api/skus/{skuId}`
- `GET /open-api/products/saleable`

说明：

- 开放接口仅返回渠道已获授权且全局可售的商品集合。
- 渠道价格、渠道限额由渠道服务聚合后输出。

### 12.3 内部服务 API

- `GET /internal/products/skus/{skuId}/saleability`
- `GET /internal/products/skus/{skuId}/snapshot`
- `GET /internal/products/skus/{skuId}/supplier-candidates`
- `POST /internal/products/skus/batch-query`
- `GET /internal/products/{productId}/rules`

## 13. 数据模型设计

### 13.1 建议表

| 表名 | 用途 |
|---|---|
| `product_categories` | 商品分类 |
| `products` | SPU 主表 |
| `product_skus` | SKU 主表 |
| `sku_supplier_mappings` | SKU 候选供应商静态映射 |
| `product_change_logs` | 商品与映射变更审计日志 |

### 13.2 关键字段要求

- `products` 必须保留商品类型、交付方式、目标类型、全局状态、有效期。
- `product_skus` 必须保留 SPU 关联、面额、运营商、地区、状态、基础价格。
- `sku_supplier_mappings` 必须保留 `skuId`、`supplierId`、`supplierSkuCode`、优先级、权重、有效期、成本价、状态。
- `product_change_logs` 必须保留操作前后值与操作者信息。

## 14. 跨服务协作

### 14.1 与订单服务

- 商品服务提供 SKU 全局可售结果与快照。
- 订单服务负责结合渠道与风控结果创建订单。

### 14.2 与渠道服务

- 商品服务提供标准商品与全局上架事实。
- 渠道服务负责商品授权、渠道价格与限额。

### 14.3 与供应商服务

- 商品服务提供静态映射。
- 供应商服务负责实际路由与协议适配。

### 14.4 与结算与账务服务

- 商品服务提供商品类型、基础面额与基础成本参考。
- 结算与账务服务基于订单成交价、成本快照与分润规则做账。

## 15. 异常处理

### 15.1 商品已下架但渠道仍发起下单

- 返回商品不可售错误。
- 记录渠道请求日志，便于运营排查是否存在缓存延迟。

### 15.2 SKU 无有效供应商映射

- 返回不可售。
- 标记为配置缺失，进入商品运维告警。

### 15.3 映射冲突

- 同一 SKU 出现多条主路由或互斥时间窗时，禁止发布生效。
- 后台需阻断保存或进入待修正状态。

### 15.4 历史订单引用已失效商品

- 允许查询历史快照。
- 不允许因当前商品状态变化而篡改历史订单解释。

## 16. 权限、审计与非功能要求

### 16.1 权限要求

- 商品运营可维护分类、SPU、SKU、映射。
- 财务角色只读商品成本参考，不具备编辑权限。
- 高风险操作如批量下架、批量停用映射需记录审计。

### 16.2 非功能要求

- SKU 查询与快照接口平均响应小于 200ms。
- 审计日志、快照数据、变更记录必须长期保留。
- 批量导出应支持异步任务化。

## 17. 风险点与控制策略

| 风险 | 影响 | 控制策略 |
|---|---|---|
| 商品定义不统一 | 下单失败、履约错误 | 强制标准 SPU/SKU 模型 |
| 映射配置错误 | 供应商路由失败 | 映射审核、冲突校验、审计日志 |
| 商品变更影响历史订单 | 对账困难 | 订单快照固化 |
| 渠道价格与商品价格混用 | 账务口径混乱 | 明确基础价格与渠道价格边界 |

## 18. 验收标准

### 18.1 功能验收

- 可完成分类、SPU、SKU、映射的完整配置闭环。
- 可输出订单所需的完整商品快照。
- 可支持一个 SKU 对接多个供应商映射。

### 18.2 数据验收

- 商品、SKU、映射、审计日志结构完整。
- 历史订单引用的商品快照可稳定查询。

### 18.3 技术验收

- 内部查询接口稳定、可追踪、带审计。
- 商品上下架与映射变更具备权限控制和日志留痕。

## 19. V1 边界

### 19.1 V1 必做

- 标准商品与 SKU 模型
- 全局上下架
- 候选供应商静态映射
- 商品快照输出
- 审计日志

### 19.2 V1 暂不做

- 复杂促销引擎
- 动态库存编排
- 复杂多维营销标签系统
- 面向终端用户的推荐与搜索排序算法
