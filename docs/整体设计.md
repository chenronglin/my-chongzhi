## 2. 项目背景与建设目标

### 2.1 项目背景

本平台是一个面向商户、代理、子代理、企业客户的交易平台，连接上游供应商与下游渠道，提供统一的商品售卖、订单处理、支付、履约、通知、分润、账务与风控能力。

平台的业务本质不是单一商城，而是一个由多个服务协作完成交易闭环的交易中台：

- 上游侧需要支持多供应商接入、协议屏蔽、可用性切换与履约容灾。
- 下游侧需要支持渠道分层、授权、价格、限额、签名鉴权与回调配置。
- 平台侧需要保证订单状态清晰、资金归口统一、异常可追溯、重试可补偿。

### 2.2 核心建设目标

1. 稳定接单：支持高并发创建订单、支付回调、供应商异步处理与通知重试。
2. 稳定供货：同一 SKU 支持多个供应商，允许主备切换、权重路由与熔断降级。
3. 渠道可扩展：支持商户、代理、子代理、企业客户等不同层级主体。
4. 资金清晰：所有收款、扣款、分润、退款、结算、对账统一归入结算与账务服务。
5. 风控可落地：同步风控拦截、异步风险信号、黑白名单、频控与人工审核闭环。
6. 运维可追踪：核心交易事件、回调、任务、资金流水都具备审计与追溯能力。

## 3. 全局设计原则

### 3.1 服务边界清晰

- 订单服务是订单主事实源，只负责状态机与事件。
- 支付服务只负责支付交易与支付回调，不判定订单最终完成。
- 供应商服务只负责适配、路由执行、状态标准化，不直接写订单主表。
- 结算与账务服务是资金唯一归口，不允许账务逻辑散落在其他服务。
- 渠道服务负责渠道主体、API Key、授权、价格、限额、白黑名单。
- 用户与权限服务负责后台账号、RBAC、数据权限与操作审计，不与渠道主体混用。
- Worker 任务系统只负责异步调度、重试与补偿，不承载业务规则定义。

### 3.2 配置与执行分离

- 商品服务维护标准商品与静态供应商映射。
- 渠道服务维护渠道授权、渠道价格、配额与回调配置。
- 供应商服务执行运行时路由、供应商协议调用与状态标准化。
- Worker 任务系统承接异步提交、轮询、重试、对账等动作。

### 3.3 事件驱动与幂等优先

- 所有跨服务关键动作通过标准事件或内部命令驱动。
- 创建订单、支付回调、供应商回调、退款、通知重试、人工补偿都必须幂等。
- 幂等键必须在文档层明确，不允许实现阶段临时决定。

### 3.4 快照固化

订单创建时必须固化以下快照：

- 商品快照
- SKU 快照
- 渠道快照
- 价格快照
- 回调配置快照
- 供应商候选映射快照

历史订单不因配置变更而改变业务解释。

### 3.5 最终一致与可补偿

- 订单、支付、供应商、通知、账务允许短暂最终一致。
- 所有异步流程必须具备重试、补偿、人工介入与审计能力。
- 终态需要保护，晚到事件只允许记录异常，不允许无规则逆转。

## 6. 总体业务架构

### 6.1 平台业务链路

下游渠道通过开放 API 发起交易，平台完成鉴权、商品校验、风控、订单创建、支付、供应商履约、账务处理、通知回调，形成完整交易闭环。

主链路如下：

1. 渠道签名请求进入 API Gateway。
2. 渠道服务校验渠道身份、授权、价格、限额与白黑名单。
3. 商品服务校验商品与 SKU 是否可售，并输出订单快照所需数据。
4. 风控服务执行同步预检。
5. 订单服务创建订单并写入事件。
6. 支付服务处理支付交易或余额扣款。
7. Worker 任务系统驱动供应商履约任务。
8. 供应商服务完成 submit/query/refund/callback 等适配动作。
9. 订单服务接收支付、供应商、通知、退款事件并推进状态。
10. 结算与账务服务统一处理收款、扣款、分润、退款、结算与对账。
11. 通知服务向渠道发起回调，并跟踪重试结果。

### 6.2 服务交互视图

```text
渠道请求
  -> API Gateway
  -> 渠道服务
  -> 商品服务
  -> 风控服务
  -> 订单服务
  -> 支付服务
  -> Worker 任务系统
  -> 供应商服务
  -> 订单服务
  -> 结算与账务服务
  -> 通知服务
```

## 9. 标准状态与标准事件

### 9.1 订单标准状态

订单采用“主状态 + 子状态 + 事件”模式。

主状态：

- `CREATED`
- `PENDING_PAYMENT`
- `PAID`
- `WAIT_SUPPLIER_SUBMIT`
- `SUPPLIER_PROCESSING`
- `SUCCESS`
- `FAIL`
- `REFUNDING`
- `REFUNDED`
- `CLOSED`

子状态至少包括：

- 支付子状态：`UNPAID`、`PAYING`、`PAID`、`PAY_FAIL`
- 供应商子状态：`UNSUBMITTED`、`ACCEPTED`、`PROCESSING`、`SUCCESS`、`FAIL`、`UNKNOWN`
- 通知子状态：`PENDING`、`SUCCESS`、`FAIL`、`RETRYING`
- 风控子状态：`PASS`、`REVIEW`、`REJECT`

### 9.2 支付与供应商标准状态

支付状态：

- `INIT`
- `PENDING`
- `SUCCESS`
- `FAIL`
- `CLOSED`
- `REFUNDING`
- `REFUNDED`

供应商标准状态：

- `ACCEPTED`
- `PROCESSING`
- `SUCCESS`
- `FAIL`
- `REFUNDING`
- `REFUNDED`
- `UNKNOWN`

### 9.3 标准事件清单

| 事件名 | 生产者 | 消费者 | 说明 | 建议幂等键 |
|---|---|---|---|---|
| `OrderCreated` | 订单服务 | 支付、风控、Worker | 订单创建完成 | `orderNo` |
| `RiskDecisionMade` | 风控服务 | 订单服务 | 风控预检或异步复核结果 | `orderNo + riskStage` |
| `PaymentSucceeded` | 支付服务 | 订单服务、结算与账务服务 | 支付成功 | `paymentNo` |
| `PaymentFailed` | 支付服务 | 订单服务 | 支付失败 | `paymentNo + status` |
| `SupplierSubmitRequested` | 订单服务 / Worker | 供应商服务 | 请求发起履约 | `orderNo + routeVersion` |
| `SupplierAccepted` | 供应商服务 | 订单服务 | 供应商已受理 | `supplierOrderNo + status` |
| `SupplierSucceeded` | 供应商服务 | 订单服务、结算与账务服务 | 供应商履约成功 | `supplierOrderNo + status` |
| `SupplierFailed` | 供应商服务 | 订单服务、结算与账务服务 | 供应商履约失败 | `supplierOrderNo + status` |
| `RefundRequested` | 订单服务 / 结算与账务服务 | 支付、供应商、Worker | 发起退款 | `orderNo + refundNo` |
| `RefundSucceeded` | 支付服务 / 供应商服务 | 订单服务、结算与账务服务 | 退款成功 | `refundNo` |
| `NotificationRequested` | 订单服务 | 通知服务 | 触发下游通知 | `orderNo + notifyType + notifyVersion` |
| `NotificationSucceeded` | 通知服务 | 订单服务 | 通知成功 | `notifyTaskNo + status` |
| `NotificationFailed` | 通知服务 | 订单服务、Worker | 通知失败待重试 | `notifyTaskNo + attemptNo` |
| `SettlementTriggered` | 订单服务 / 支付服务 | 结算与账务服务 | 触发入账、分润或冲正 | `orderNo + actionType` |

## 10. 核心业务流程

### 10.1 鉴权下单流程

1. 渠道通过 `AccessKey + Sign + Timestamp + Nonce` 调用开放 API。
2. 渠道服务校验渠道状态、签名版本、IP 白名单、配额与商品授权。
3. 商品服务返回 SKU 可售结果、商品快照、供应商候选映射。
4. 风控服务执行同步风控校验。
5. 订单服务创建订单，写入 `OrderCreated` 事件。
6. 若需要支付，支付服务创建支付单并返回支付信息。
7. 若为余额即时扣款或免支付订单，直接进入待履约状态。

### 10.2 支付回调流程

1. 支付渠道回调支付服务。
2. 支付服务校验签名、金额、支付状态并做幂等处理。
3. 支付服务写支付回调日志并发布 `PaymentSucceeded` 或 `PaymentFailed`。
4. 订单服务据此推进订单状态。
5. 结算与账务服务据此登记入账、冻结或冲正动作。

### 10.3 供应商履约流程

1. Worker 任务系统根据订单状态投递履约任务。
2. 供应商服务读取订单快照中的候选路由，执行适配器。
3. 同步返回受理则写 `SupplierAccepted`。
4. 若供应商异步回调，则通过回调接口解析标准状态。
5. 若供应商未回调或状态不明，则由 Worker 发起轮询。
6. 订单服务接收标准化供应商事件，推进成功、失败或退款流程。

### 10.4 退款与冲正流程

1. 订单失败且已收款、人工退款、渠道退款等场景触发退款。
2. 订单服务进入 `REFUNDING`，并发布 `RefundRequested`。
3. 支付服务或供应商服务分别执行退款。
4. 结算与账务服务冲销原流水并记录退款流水。
5. 订单服务接收退款结果，推进至 `REFUNDED` 或保留异常待处理。

### 10.5 通知重试流程

1. 订单终态或关键状态变化触发 `NotificationRequested`。
2. 通知服务按回调配置签名并发送。
3. 投递成功则反馈 `NotificationSucceeded`。
4. 失败则生成重试计划，由 Worker 任务系统按退避策略执行。
5. 达到阈值后进入死信并支持人工补发。

## 11. 接口与集成规范

### 11.1 接口分类与路径约定

- 后台管理 API：`/admin/**`
- 开放平台 API：`/open-api/**`
- 内部服务 API：`/internal/**`
- 第三方回调 API：`/callbacks/**`

### 11.2 鉴权规范

后台接口：

- 采用登录 Token 鉴权。
- 敏感操作可叠加二次校验或审批。

开放接口：

- 采用 `AccessKey + Sign + Timestamp + Nonce`。
- 必须防重放、校验时钟偏差、限制来源 IP。

内部接口：

- 采用服务间鉴权凭证、来源服务标识与请求签名。

### 11.3 幂等规范

以下场景必须幂等：

- 创建订单
- 支付回调
- 供应商回调
- 退款申请
- 通知重试
- Worker 重放与人工补偿

建议幂等键：

- `channelId + channelOrderNo`
- `paymentNo`
- `supplierOrderNo`
- `refundNo`
- `notifyTaskNo + attemptNo`
- `jobType + businessKey`

### 11.4 返回体规范

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "requestId": "9f5d31f4b5f4"
}
```

### 11.5 错误码分层

- `1xxx` 参数错误
- `2xxx` 鉴权错误
- `3xxx` 权限错误
- `4xxx` 渠道与商品错误
- `5xxx` 订单错误
- `6xxx` 支付错误
- `7xxx` 供应商错误
- `8xxx` 结算与账务错误
- `9xxx` 风控、通知、任务与系统错误

### 11.6 Trace 与审计

- 所有接口、任务、事件必须携带 `requestId` / `traceId`。
- 支持按订单号、渠道订单号、支付单号、供应商订单号、任务号检索。
- 敏感配置变更必须记录操作人、时间、来源 IP、变更前后值。

## 12. 核心对象模型

核心对象包括：

- 后台用户 UserAccount
- 角色 Role
- 权限 Permission
- 渠道 Channel
- 商品 Product
- SKU ProductSku
- SKU 供应商映射 SkuSupplierMapping
- 订单 Order
- 订单事件 OrderEvent
- 支付单 PaymentOrder
- 供应商订单 SupplierOrder
- 账户 Account
- 账务流水 LedgerEntry
- 分润规则 ProfitRule
- 风控决策 RiskDecision
- 通知任务 NotificationTask
- Worker 任务 WorkerJob

通用设计要求：

- 所有核心对象必须包含创建时间、更新时间、创建人、更新人。
- 关键业务对象默认使用软删除或状态失效，不做物理删除。
- 订单、账务、回调、审计日志必须长期保留。

## 13. 非功能、安全与审计要求

### 13.1 性能要求

- 订单创建接口平均响应时间小于 300ms，不含异步履约。
- 商品、订单、渠道查询接口平均响应时间小于 200ms。
- 支付回调、供应商回调、通知回调需要支持高并发接入。
- Worker 任务系统需要支持水平扩展。

### 13.2 可用性与一致性要求

- 核心交易链路可用性不低于 99.9%。
- 订单、支付、供应商、账务、通知关键数据不能无感丢失。
- 账务与订单允许短暂最终一致，但必须可补偿、可审计。

### 13.3 安全要求

- API Key、支付密钥、供应商凭证必须加密存储。
- 敏感字段按角色与数据权限脱敏展示。
- 回调接口必须验签、防重放、防伪造。
- 审计日志、账务日志、任务日志必须可追溯、不可篡改。

## 14. 版本规划与实施原则

### 14.1 V1 必做

- 10 个核心服务的职责边界与接口口径统一
- 单订单主链路闭环：下单、支付、履约、账务、通知
- 1 至 2 个供应商适配器
- 余额支付与 1 个第三方支付渠道
- 基础风控、基础分润、基础对账、基础通知重试
- Worker 独立部署

### 14.2 V2 建议

- 多级代理分润增强
- 更多支付与供应商渠道
- 高级风控规则与评分模型
- 自动对账与人工补偿工具增强
- 更丰富的通知模板与渠道

### 14.3 实施原则

- 先固化领域边界，再展开接口与数据设计。
- 先确保闭环，再扩展复杂策略。
- 任何跨服务新能力都必须先确认数据归属与事件归属。

## 16. 验收标准

### 16.1 业务验收

- 能完成鉴权下单、支付成功、供应商履约、账务处理、通知回调的闭环。
- 能支持渠道禁用、商品下架、风控拦截、供应商失败、退款冲正等关键分支。
- 能查看订单全生命周期事件、支付轨迹、供应商履约记录、账务流水与通知日志。

### 16.2 文档验收

- 主文档与 10 份详细 PRD 术语、命名、事件、数据归属完全一致。
- 不保留 AI 生成前后缀、占位说明、未落实项提示。
- 每份详细 PRD 都包含职责边界、核心对象、功能明细、API、数据模型、异常处理、验收标准。

### 16.3 设计质量验收

- 订单、支付、供应商、结算与账务、通知、Worker 的事件边界可直接指导研发拆分。
- 相同概念只有一个权威定义，其他文档仅引用或摘要。
