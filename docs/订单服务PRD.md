# 《订单服务详细 PRD》

## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 文档名称 | 订单服务详细 PRD |
| 所属系统 | 交易平台后台 API 服务系统 |
| 模块名称 | 订单服务 / Order Service |
| 版本 | V1.0 |
| 优先级 | P0 |

## 2. 模块背景与定位

### 2.1 模块背景

订单服务是交易平台的状态控制中枢。平台涉及支付、供应商履约、通知、退款、账务、风控等多个服务，如果订单服务边界不清晰，常见后果包括：

- 状态定义散乱，无法判断订单真实进度。
- 支付、供应商、账务都直接改订单，造成事实冲突。
- 重复回调与晚到事件导致重复成功、重复退款。
- 问题订单无法还原完整事件轨迹。

### 2.2 一句话定位

订单服务是平台交易事实源，负责创建订单、维护状态机、记录事件、对接外部业务事件并对外提供订单查询能力。

### 2.3 模块价值

1. 统一订单事实与状态口径。
2. 统一支付、供应商、风控、通知等事件入口。
3. 为结算与账务、通知、Worker 提供标准驱动点。
4. 为运营、客服、财务提供统一查询与追溯出口。

## 3. 职责边界

### 3.1 订单服务负责什么

1. 创建订单并校验幂等。
2. 维护主状态、支付子状态、供应商子状态、通知子状态、风控子状态。
3. 记录订单全生命周期事件。
4. 提供订单详情、列表、轨迹与人工操作能力。
5. 接收支付、供应商、通知、风控、退款相关标准事件。
6. 根据状态变化触发后续异步动作指令。

### 3.2 订单服务不负责什么

1. 不接入第三方支付渠道。
2. 不执行供应商协议。
3. 不做账务记账、分润、结算与对账。
4. 不做短信、邮件、Webhook 的实际发送。
5. 不定义风控规则。
6. 不做库存、卡密、履约资源分配。

### 3.3 典型边界约束

| 场景 | 是否属于订单服务 |
|---|---|
| 创建订单 | 是 |
| 校验渠道订单号幂等 | 是 |
| 接收支付成功事件并更新订单状态 | 是 |
| 调用支付渠道下单 | 否，属于支付服务 |
| 选择运行时供应商 | 否，属于供应商服务 |
| 记录分润流水 | 否，属于结算与账务服务 |
| 发送下游回调 | 否，属于通知服务 |

## 4. 设计原则

### 4.1 单一事实源原则

订单主表和订单事件表只能由订单服务写入，其他服务不得直接修改订单主数据。

### 4.2 状态驱动原则

支付、供应商、通知、退款等动作都以订单状态与订单事件为驱动，不以临时标记或人工口头约定驱动。

### 4.3 事件可追溯原则

每次状态变化都必须留下事件，事件需要包含来源、前后状态、业务原因与操作者。

### 4.4 幂等优先原则

创建、回调、重试、补偿、人工操作都必须定义幂等键。

### 4.5 终态保护原则

`SUCCESS`、`FAIL`、`REFUNDED`、`CLOSED` 为受保护终态。晚到事件默认只做审计与异常标记，不自动反向覆盖。

## 5. 模块目标与成功标准

### 5.1 业务目标

- 支撑所有渠道的统一订单创建与查询。
- 形成清晰、可追溯、可恢复的交易状态链路。
- 让下游、运营、客服、财务都能基于统一订单事实工作。

### 5.2 技术目标

- 高并发订单创建。
- 标准事件接入。
- 回调幂等。
- 状态流转可审计。
- 支持异步补偿与人工介入。

### 5.3 成功标准

- 订单创建成功率不低于 99.9%。
- 每个订单都能查询到完整状态轨迹。
- 重复回调不产生重复成功或重复退款。
- 终态订单不会被普通晚到事件非法覆盖。

## 7. 核心对象与字段

### 7.1 订单定义

订单是平台交易事实记录，代表渠道针对某个标准 SKU 发起的一次交易请求。

V1 约束：

- 一张订单对应一个 SKU。
- 不支持拆单与组合订单。
- 支持支付与免支付两类订单。

### 7.2 订单关键字段

标识类：

- `orderNo`
- `channelOrderNo`
- `channelId`
- `parentChannelId`

商品类：

- `productId`
- `skuId`
- `productSnapshot`
- `supplierRouteSnapshot`

交易类：

- `salePrice`
- `costPrice`
- `currency`
- `paymentMode`
- `paymentNo`

状态类：

- `mainStatus`
- `paymentStatus`
- `supplierStatus`
- `notifyStatus`
- `riskStatus`

时间类：

- `createdAt`

审计类：

- `remark`

## 8. 状态机设计

### 8.1 主状态

- `CREATED`
- `PENDING_PAYMENT`
- `PAID`
- `WAIT_SUPPLIER_SUBMIT`
- `SUPPLIER_PROCESSING`
- `SUCCESS`
- `FAIL`
- `REFUNDING`
- `REFUNDED`
- `CLOSED`

### 8.2 子状态

支付子状态：

- `UNPAID`
- `PAYING`
- `PAID`
- `PAY_FAIL`

供应商子状态：

- `UNSUBMITTED`
- `ACCEPTED`
- `PROCESSING`
- `SUCCESS`
- `FAIL`
- `UNKNOWN`

通知子状态：

- `PENDING`
- `SUCCESS`
- `FAIL`
- `RETRYING`

风控子状态：

- `PASS`
- `REVIEW`
- `REJECT`

### 8.3 标准主链路

```text
CREATED
  -> PENDING_PAYMENT
  -> PAID
  -> WAIT_SUPPLIER_SUBMIT
  -> SUPPLIER_PROCESSING
  -> SUCCESS / FAIL
  -> REFUNDING（若已收款且需退款）
  -> REFUNDED
```

### 8.4 终态与异常原则

- `SUCCESS` 后再收到供应商失败，不自动回滚，进入异常待处理。
- `CLOSED` 后再收到支付成功，只记录异常，不自动开单。
- `REFUNDED` 后再收到供应商成功，标记高风险异常，转人工处理。

## 9. 事件模型设计

### 9.1 事件分类

- 订单创建事件
- 支付事件
- 供应商事件
- 风控事件
- 通知事件
- 退款事件
- 人工操作事件
- Worker 补偿事件

### 9.2 事件标准字段

建议所有订单事件统一包含：

- `eventNo`
- `orderNo`
- `eventType`
- `sourceService`
- `sourceNo`
- `beforeStatus`
- `afterStatus`
- `payload`
- `operator`
- `occurredAt`
- `requestId`

### 9.3 事件使用规则

1. 状态变化必须对应事件。
2. 同一业务事件重复到达时只能产生一次有效推进。
3. 终态异常事件也要落库，不允许直接丢弃。

## 10. 核心流程设计

### 10.1 创建订单

1. 渠道通过开放 API 发起创建请求。
2. 订单服务校验 `channelId + channelOrderNo` 幂等。
3. 调用商品服务获取快照与可售结果。
4. 调用渠道服务获取渠道价格、授权与回调配置。
5. 调用风控服务做同步预检。
6. 写订单主表与 `OrderCreated` 事件。
7. 根据支付模式进入待支付或待履约状态。

### 10.2 支付事件接入

1. 支付服务回传支付结果事件。
2. 订单服务按 `paymentNo` 幂等处理。
3. 支付成功则推进到 `PAID` 或 `WAIT_SUPPLIER_SUBMIT`。
4. 支付失败则视规则进入 `CLOSED` 或保留可重试状态。

### 10.3 供应商事件接入

1. 供应商服务回传受理、处理中、成功、失败、退款事件。
2. 订单服务按 `supplierOrderNo + status` 幂等处理。
3. 同步受理进入 `SUPPLIER_PROCESSING`，终态结果进入 `SUCCESS` 或 `FAIL`。

### 10.4 通知事件接入

1. 订单状态进入需要通知的关键节点。
2. 通知服务回传通知成功、失败、重试中结果。
3. 订单服务仅维护通知子状态，不负责投递。

### 10.5 人工操作

支持：

- 人工关闭
- 标记异常
- 追加备注
- 触发重新通知
- 触发重新提交供应商
- 触发重新轮询

不支持：

- 手工改成功
- 手工改已退款为成功
- 直接跳过账务结果改终态

## 11. 功能需求明细

### 11.1 订单创建

- 支持开放 API 创建订单。
- 支持立即返回支付信息或已创建结果。
- 必须写入渠道、商品、价格、回调配置、候选供应商等快照。

### 11.2 订单查询

- 支持按 `orderNo`、`channelOrderNo`、`channelId` 查询。
- 支持订单详情、状态、快照、事件轨迹查询。
- 支持后台分页、筛选与导出。

### 11.3 订单轨迹

- 展示主状态变化、子状态变化、来源服务、时间、操作者。
- 支持检索支付事件、供应商事件、通知事件、人工操作。

### 11.4 超时处理

- 未支付超时可关闭。
- 提交供应商超时应进入待重试或轮询，不直接判失败。
- 通知超时由通知服务与 Worker 协同重试。

## 12. 核心规则

1. 订单主表只能由订单服务写入。
2. 创建订单必须固化快照。
3. 任何状态推进必须基于合法流转白名单。
4. 终态默认不可逆。
5. 订单服务只接收标准事件，不解析第三方协议原文。
6. 订单失败且已收款时，必须进入退款或人工异常流程，不允许直接静默失败。

## 14. 数据模型设计

### 14.1 建议表

| 表名 | 用途 |
|---|---|
| `orders` | 订单主表 |
| `order_events` | 订单事件表 |
| `order_remarks` | 订单备注表 |

### 14.2 关键字段要求

- `orders` 必须保留渠道快照、商品快照、价格快照、回调配置快照、候选供应商快照。
- `orders` 必须保留主状态、支付子状态、供应商子状态、通知子状态、风控子状态与版本号。
- `order_events` 必须保留事件来源、前后状态、幂等键、原始载荷摘要。

### 14.3 索引建议

- 唯一索引：`channel_id + channel_order_no`
- 普通索引：`order_no`、`channel_id + created_at`、`main_status + created_at`
- 事件索引：`order_no + occurred_at`

## 15. 跨服务协作

| 对方服务 | 订单服务依赖 | 订单服务输出 |
|---|---|---|
| 商品服务 | 可售校验、商品快照、候选供应商快照 | 下单结果引用 |
| 渠道服务 | 渠道校验、渠道价格、授权、回调配置 | 订单结果供渠道查询 |
| 支付服务 | 支付结果事件、退款结果 | 支付创建指令、支付状态消费 |
| 供应商服务 | 受理、处理中、成功、失败、退款事件 | 履约指令与查询上下文 |
| 结算与账务服务 | 无直接写入依赖 | 订单成功、失败、退款事件 |
| 通知服务 | 通知结果事件 | 通知指令 |
| 风控服务 | 风控决策事件 | 订单上下文 |
| Worker 任务系统 | 异步执行结果 | 提交、轮询、重试命令 |

## 16. 异常处理

### 16.1 重复支付回调

- 首次有效成功后，重复回调仅记录事件，不重复推进状态。

### 16.2 重复供应商回调

- 相同状态重复事件忽略。
- 不同终态冲突时标记异常并转人工。

### 16.3 已关闭订单收到支付成功

- 不自动恢复订单。
- 记录异常事件，并触发人工核对与可能退款流程。

### 16.4 已退款订单收到供应商成功

- 标记高风险异常。
- 由运营、财务、供应商联合处理。

## 17. 权限、审计与非功能

### 17.1 权限设计

- 开放接口按渠道身份和签名鉴权。
- 后台接口按角色与数据权限控制。
- 敏感字段如成本价、回调原文、异常标签按权限脱敏。

### 17.2 非功能要求

- 订单创建平均响应小于 300ms。
- 查询接口平均响应小于 200ms。
- 支持高并发事件接入与状态更新。

### 17.3 审计要求

- 人工操作必须记录操作者、原因、原状态、目标动作、时间、IP。
- 支持按 `orderNo`、`channelOrderNo`、`requestId`、`paymentNo`、`supplierOrderNo` 检索。

## 18. 风险点与控制策略

| 风险 | 影响 | 控制策略 |
|---|---|---|
| 订单状态混乱 | 无法对账、无法退款 | 明确状态机与合法流转白名单 |
| 终态被晚到事件覆盖 | 资金与履约冲突 | 终态保护与异常标记 |
| 快照不全 | 无法追责与复盘 | 强制快照字段 |
| 重复回调 | 重复成功或重复退款 | 幂等键与事件去重 |

## 19. 验收标准

### 19.1 功能验收

- 支持创建订单、查询详情、查询轨迹、人工操作。
- 支持支付、供应商、通知、退款事件接入。
- 支持异常标记和终态保护。

### 19.2 数据验收

- 订单主表、事件表、备注表字段完整。
- 任意订单都可追溯完整状态轨迹。

### 19.3 技术验收

- 创建与事件接入具备幂等能力。
- 状态推进具备版本控制或同类并发保护能力。
- 关键操作具备 Trace 与审计日志。

## 20. V1 边界

### 20.1 V1 必做

- 单订单单 SKU 模型
- 主状态 + 子状态 + 事件日志
- 幂等创建
- 支付、供应商、通知、退款事件接入
- 异常标记与人工关闭

### 20.2 V1 暂不做

- 组合订单
- 拆单
- 可视化状态机编排
- 全量事件重演恢复
