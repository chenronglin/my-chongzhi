# 《支付服务详细 PRD》

## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 文档名称 | 支付服务详细 PRD |
| 所属系统 | 交易平台后台 API 服务系统 |
| 模块名称 | 支付服务 / Payment Service |
| 版本 | V1.0 |
| 优先级 | P0 |
| 文档目标 | 明确支付交易、支付渠道、支付回调、余额支付与退款指令边界 |

## 2. 模块背景与定位

### 2.1 模块背景

支付是交易闭环中最敏感的链路之一。若支付能力分散在订单或结算与账务服务中，容易造成状态混乱与资金责任不清。

### 2.2 一句话定位

支付服务是平台支付交易服务，负责支付单、支付渠道、支付回调、余额支付与退款指令，但不负责订单最终成功判定。

### 2.3 模块价值

1. 统一第三方支付与余额支付入口。
2. 统一支付回调处理与幂等。
3. 为订单服务提供稳定支付事件。
4. 为结算与账务服务提供支付事实记录。

## 3. 职责边界

### 3.1 支付服务负责什么

1. 支付单创建与状态管理。
2. 支付渠道配置管理。
3. 支付回调接收、验签与幂等处理。
4. 余额支付执行。
5. 支付状态查询与退款指令发起。

### 3.2 支付服务不负责什么

1. 不控制订单最终主状态。
2. 不做分润、账务、结算与对账。
3. 不做供应商履约。
4. 不决定渠道价格。

## 4. 设计原则

### 4.1 支付事实独立原则

支付单必须独立于订单主表建模，支付结果通过标准事件通知订单服务。

### 4.2 回调幂等优先原则

支付回调是高风险入口，必须先验签、再幂等、后推进状态。

### 4.3 一笔订单单一有效支付原则

V1 一张订单只允许一个有效支付成功结果。

### 4.4 退款指令与退款结果分离原则

支付服务负责发起退款与接收退款结果，订单与结算根据标准事件推进。

## 5. 模块目标与成功标准

### 5.1 业务目标

- 支持主流第三方支付与余额支付。
- 保证支付状态清晰、回调稳定、退款可追踪。

### 5.2 技术目标

- 高并发回调接入。
- 支付事件幂等。
- 支付链路可审计与可检索。

### 5.3 成功标准

- 支付成功不会重复推进订单。
- 金额不匹配或签名异常能被及时拦截。
- 退款动作可追踪到支付单与订单。

## 6. 角色与使用场景

| 角色 | 使用场景 |
|---|---|
| 渠道 / 商户 | 获取支付信息、查询支付结果 |
| 平台财务 | 查询支付单、退款状态 |
| 支付运营 | 维护支付渠道与配置 |
| 订单服务 | 发起支付、消费支付结果 |
| 结算与账务服务 | 记录收款、退款资金事实 |

- `DISABLED`

## 9. 核心流程设计

### 9.1 支付创建流程

1. 订单服务发起支付创建。
2. 支付服务校验订单、金额、支付方式。
3. 创建支付单。
4. 返回支付参数、二维码、收银台地址或余额支付结果。

### 9.2 支付回调流程

1. 第三方支付回调支付服务。
2. 验签、校验金额与交易状态。
3. 幂等处理后更新支付单。
4. 发布支付成功或失败事件。

### 9.3 退款流程

1. 订单服务或结算与账务服务发起退款请求。
2. 支付服务创建退款记录并调用支付渠道。
3. 接收退款结果或轮询确认结果。
4. 输出标准退款事件。

## 10. 功能需求明细

### 10.1 支付渠道管理

- 支持微信、支付宝、余额等渠道配置。
- 支持启停、维护状态、商户号、证书、回调配置。

### 10.2 支付单管理

- 创建支付单。
- 查询支付单状态。
- 关联订单、渠道、支付方式。

### 10.3 余额支付

- 调用结算与账务服务校验可用余额。
- 余额扣款成功后直接输出支付成功事件。

### 10.4 回调处理

- 必须验签。
- 必须幂等。
- 必须记录回调原始摘要。

### 10.5 退款指令

- 支持全额退款。
- V1 不支持复杂拆分退款。

## 11. 核心规则

1. 一个支付单只允许一个有效成功结果。
2. 支付服务不直接修改订单主状态。
3. 支付金额必须与订单待支付金额一致。
4. 回调异常时优先保留日志与人工可追溯性。
5. 余额支付的资金扣减由结算与账务服务负责记账。

## 14. 跨服务协作

### 14.1 与订单服务

- 订单服务发起支付创建并接收支付结果事件。

### 14.2 与结算与账务服务

- 支付成功、余额扣减、退款结果作为资金事实输入。

### 14.3 与渠道服务

- 可根据渠道策略限制支付方式。

## 15. 异常处理

### 15.1 金额不匹配

- 拒绝受理。
- 记录高风险回调日志并告警。

### 15.2 重复回调

- 幂等忽略重复成功。
- 保留重复回调日志。

### 15.3 已关闭订单收到支付成功

- 输出支付成功异常事件给订单服务，由订单服务决定后续流程。

### 15.4 退款异常

- 退款结果不明时进入查询补偿流程。

## 16. 权限、审计与非功能要求

### 16.1 权限要求

- 支付渠道配置、证书更新、退款重试为高风险操作，需重点审计。

### 16.2 非功能要求

- 回调接口支持高并发。
- 支付单查询平均响应小于 150ms。
- 敏感证书与密钥需加密存储。

## 17. 风险点与控制策略

| 风险 | 影响 | 控制策略 |
|---|---|---|
| 回调重复 | 重复推进订单 | 回调幂等 |
| 金额校验缺失 | 资金风险 | 金额强校验 |
| 支付结果与订单脱节 | 状态不一致 | 标准事件驱动 |
| 退款状态不明 | 账务悬挂 | 退款查询补偿 |

## 18. 验收标准

### 18.1 功能验收

- 可完成支付单创建、回调处理、余额支付、退款指令。

### 18.2 数据验收

- 支付单、回调日志、退款记录完整可查。

### 18.3 技术验收

- 回调具备幂等、验签、金额校验。
- 支付结果可通过标准事件对接订单服务。

## 19. V1 边界

### 19.1 V1 必做

- 余额支付
- 1 个第三方支付渠道
- 支付回调
- 全额退款

### 19.2 V1 暂不做

- 多笔合并支付
- 分单退款
- 支付营销活动编排
