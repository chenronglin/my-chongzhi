# 当前后端系统已实现业务功能说明

本文档从业务视角说明当前后端程序代码**已经实现并可运行验证**的业务功能，用于与业务部门确认当前 V1 阶段的实际落地范围。

说明口径：

- 本文档只描述**代码中已经实现**的能力。
- 本文档不描述尚未开发完成、仅在 PRD 中规划但代码中未落地的能力。
- 对于已经实现但目前仍为**模拟通道 / 占位实现**的功能，会明确标注“当前边界”。

## 1. 当前系统整体能力概览

当前后端已经落地为一个可运行的 API 服务系统，已完成以下 10 个业务模块的基础实现：

1. 用户与权限服务
2. 渠道服务
3. 商品服务
4. 风控服务
5. 订单服务
6. 支付服务
7. 供应商服务
8. 结算与账务服务
9. 通知服务
10. Worker 任务系统

当前已经打通一条可以运行验证的核心业务闭环：

**渠道签名下单 -> 商品校验 -> 渠道校验 -> 风控预检 -> 创建订单 -> 余额支付 -> Worker 提交供应商 -> 供应商履约成功 -> 账务处理 -> Webhook 通知成功**

## 2. 已实现的业务功能

### 2.1 后台账号与权限

当前已实现：

- 后台管理员账号登录。
- 登录后签发 JWT 访问令牌。
- Refresh Token 刷新登录态。
- 后台退出登录。
- 后台用户查询。
- 后台用户新增。
- 角色查询。
- 角色新增。
- 后台操作审计日志写入能力。

当前适合业务确认的点：

- 后台是否只区分“内部管理账号”，不与商户/渠道账号混用。
- 当前是否接受以“超级管理员 + 若干基础角色”的方式先启动。

### 2.2 渠道管理

当前已实现：

- 渠道主体新增与查询。
- 渠道 API Key / SecretKey 配置。
- 开放接口签名鉴权：
  - `AccessKey`
  - `Sign`
  - `Timestamp`
  - `Nonce`
- 渠道商品授权配置。
- 渠道价格配置。
- 渠道单笔 / 日 / 月 / QPS 限额配置。
- 渠道回调地址与回调签名秘钥配置。
- 下单前渠道策略校验：
  - 渠道是否有效
  - 商品是否已授权
  - 订单金额是否超单笔限额
  - 是否已配置回调地址

当前适合业务确认的点：

- 当前 V1 是否接受“渠道禁用、未授权、超限时直接拒单”的处理方式。
- 当前是否接受“渠道价格”按 SKU 维度管理。

### 2.3 商品中心

当前已实现：

- 商品分类新增与查询。
- 商品新增与查询。
- SKU 新增。
- SKU 与供应商静态映射配置。
- 商品列表开放查询。
- 下单前 SKU 全局可售校验。
- 订单创建所需商品快照输出：
  - 商品快照
  - SKU 快照
  - 候选供应商快照

当前业务规则已落地：

- 商品必须处于可售状态。
- SKU 必须处于上架状态。
- SKU 必须存在有效供应商映射。
- 商品服务只负责**全局可售**，不负责渠道授权判断。

当前适合业务确认的点：

- 当前 V1 是否接受先按“标准商品 + SKU + 静态映射”运行，不做复杂促销和动态库存。

### 2.4 风控

当前已实现：

- 下单前同步风控预检。
- 黑名单 / 白名单基础能力。
- 金额规则基础能力。
- 风控决策落库。
- 风控规则查询与新增。

当前风控决策结果支持：

- `PASS`：通过
- `REVIEW`：人工复核
- `REJECT`：拒绝

当前业务规则已落地：

- 渠道白名单优先放行。
- 渠道黑名单直接拒绝。
- IP 黑名单直接拒绝。
- 金额超过阈值时进入人工复核。

当前适合业务确认的点：

- 当前 V1 是否接受“只有基础黑白名单 + 金额预警”这一版风控策略。
- 当前是否接受“人工复核仅保留结果，不做完整人工审核工作台”。

### 2.5 订单中心

当前已实现：

- 开放接口创建订单。
- 根据 `channelId + channelOrderNo` 做下单幂等。
- 订单详情查询。
- 订单事件轨迹查询。
- 后台订单列表查询。
- 后台人工关闭订单。
- 后台标记异常。
- 后台追加备注。
- 订单事件日志落库。

当前订单状态主链路已实现：

- `PENDING_PAYMENT`
- `WAIT_SUPPLIER_SUBMIT`
- `SUPPLIER_PROCESSING`
- `SUCCESS`
- `REFUNDING`
- `REFUNDED`
- `CLOSED`

当前订单子状态已实现并在状态推进中使用：

- 支付状态
- 供应商状态
- 通知状态
- 风控状态

当前业务规则已落地：

- 下单时固化渠道快照、商品快照、回调配置快照、供应商候选快照、风控快照。
- 每次状态变化必须记录事件。
- 同一渠道订单号重复提交时返回原订单。
- 供应商成功后订单进入成功。
- 供应商失败且已支付时订单进入退款流程。
- 退款成功后订单进入已退款。

当前适合业务确认的点：

- 当前 V1 是否接受“单订单单 SKU”模型。
- 当前是否接受“终态默认不可逆，异常靠人工标记处理”。

### 2.6 支付

当前已实现：

- 支付通道查询。
- 支付单查询。
- 内部创建支付单。
- 余额支付。
- 模拟第三方支付回调。
- 支付退款记录。
- 退款成功事件输出。

当前已实现的支付方式：

- `BALANCE`：余额支付
- `ONLINE`：模拟在线支付通道 `mockpay`
- `FREE`：免支付

当前业务规则已落地：

- 一张支付单只允许一个有效成功结果。
- 余额支付成功后立即推进订单。
- 模拟在线支付通过回调推进订单。
- 支付退款成功后进入退款成功事件链路。

当前适合业务确认的点：

- 当前 V1 是否接受先只用“余额支付 + 模拟在线支付”完成业务验证。
- 当前是否接受“真实微信/支付宝后续再接入”。

### 2.7 供应商履约

当前已实现：

- 供应商主数据查询与新增。
- 供应商配置保存。
- Worker 异步提交供应商。
- Worker 异步查询供应商结果。
- 供应商回调接收。
- 供应商订单记录。
- 供应商回调日志记录。

当前已实现的履约方式：

- 默认模拟供应商 `mock-supplier`
- 提交后先进入“已受理”
- 再由异步查询推进为成功或失败

当前业务规则已落地：

- 供应商服务不直接修改订单主表。
- 供应商结果通过标准事件驱动订单状态变化。
- 支持通过订单扩展字段模拟供应商失败场景。

当前适合业务确认的点：

- 当前 V1 是否接受先用模拟供应商完成履约流程验证。
- 当前是否接受“真实供应商协议接入后续逐家补充”。

### 2.8 结算与账务

当前已实现：

- 账户查询。
- 账务流水查询。
- 分润规则查询与新增。
- 平台账户与渠道账户基础模型。
- 余额支付扣减渠道余额并增加平台余额。
- 订单成功时进行基础结算记账。
- 退款成功时进行资金冲正。

当前业务规则已落地：

- 所有资金变化都写流水。
- 相同引用动作不会重复记账。
- 退款会走冲正链路。

当前适合业务确认的点：

- 当前 V1 是否接受“先完成基础记账与退款冲正”，复杂结算单与对账流程后续增强。

### 2.9 通知

当前已实现：

- 订单成功 / 退款成功触发通知任务。
- Webhook 通知任务创建。
- 通知签名生成。
- 通知投递日志记录。
- 通知失败重试。
- 通知死信记录。
- 后台通知任务与死信查询。
- 后台手工重试入口。

当前通知投递方式：

- Webhook 为主
- 短信接口为占位实现
- 邮件接口为占位实现

当前业务规则已落地：

- 通知使用订单创建时快照下来的回调配置。
- 通知失败不会直接丢弃，而是进入重试或死信。

当前适合业务确认的点：

- 当前 V1 是否接受“Webhook 真实可用，短信/邮件先占位”。

### 2.10 Worker 异步任务系统

当前已实现：

- 异步任务入队。
- 延迟执行。
- 自动重试。
- 死信转移。
- 死信查询。
- 后台任务查询。
- 后台任务手工重试。
- 后台任务取消。

当前已接入的任务类型：

- 供应商提交任务
- 供应商查询任务
- 通知投递任务

当前适合业务确认的点：

- 当前 V1 是否接受“订单履约和通知全部通过异步任务执行”的模式。

## 3. 当前已经验证通过的业务闭环

代码中已通过测试验证的闭环为：

1. 渠道使用签名方式调用开放接口下单。
2. 系统完成渠道校验、商品校验、风控预检。
3. 订单创建成功并固化业务快照。
4. 余额支付成功。
5. 订单进入待提交供应商状态。
6. Worker 异步提交供应商。
7. 供应商受理后继续异步查询。
8. 供应商返回成功。
9. 订单变为成功。
10. 账务完成基础结算动作。
11. 通知任务生成并成功回调。
12. 订单通知状态变为成功。

这意味着：**当前系统已经具备一条可运行的“下单到完成”业务主链路。**

## 4. 当前系统边界与未完全落地项

以下能力在代码中**不是完整生产形态**，业务确认时请特别注意：

### 4.1 支付通道仍是模拟实现

- 当前在线支付使用 `mockpay` 模拟通道。
- 真实微信、支付宝等外部支付通道尚未接入。

### 4.2 供应商仍是模拟实现

- 当前履约使用 `mock-supplier` 模拟供应商。
- 真实供应商协议、签名、回调格式适配尚未接入。

### 4.3 通知通道仅 Webhook 真实可演练

- Webhook 可运行。
- 短信与邮件当前是接口桩，不是实际发送能力。

### 4.4 冻结 / 解冻能力尚未真正启用

- 结算与账务服务保留了冻结、解冻接口。
- 当前代码中仍是“预留接口”，没有真正执行冻结余额逻辑。

### 4.5 供应商退款当前由支付退款兜底

- 供应商退款接口目前是占位说明。
- 当前退款主链路主要由支付退款完成闭环。

### 4.6 高级业务能力尚未落地

当前尚未落地的典型能力包括：

- 复杂多级代理分润计算
- 完整财务结算单业务操作流
- 完整对账处理闭环
- 完整人工审核工作台
- 真实短信、邮件、第三方供应商、第三方支付接入
- 更完整的后台编辑、停用、审批、导出能力

## 5. 建议业务部门本轮重点确认事项

建议业务部门本轮重点确认以下内容：

1. 当前 V1 是否接受“先跑通一条完整交易闭环，再逐步补真实通道”。
2. 当前渠道体系、商品体系、订单状态、回调机制是否符合业务预期。
3. 当前风险策略是否接受先以“黑白名单 + 金额阈值”为主。
4. 当前账务口径是否接受先以“余额扣款、基础入账、退款冲正”为主。
5. 当前通知口径是否接受先以 Webhook 为主通道。
6. 当前系统中哪些模拟能力必须优先替换为真实能力：
   - 支付
   - 供应商
   - 通知
   - 结算

## 6. 结论

从业务可确认角度看，当前后端系统已经不是纯骨架，而是具备了：

- 可登录的后台管理能力
- 可签名访问的开放接口能力
- 可配置的渠道、商品、价格、限额、回调基础能力
- 可创建并流转的订单能力
- 可执行的支付、履约、通知、账务闭环
- 可补偿的异步任务机制

当前最适合的定位是：

**“可用于业务流程确认和联调验证的 V1 可运行版本”**

而不是：

**“所有真实外部通道都已接入完毕的生产最终版”**
